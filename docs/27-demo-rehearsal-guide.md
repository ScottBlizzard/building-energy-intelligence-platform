# 27 · 答辩演示指导脚本（Demo Rehearsal Guide）

> 目的：给"答辩/汇报现场"一份逐步可照做的演示脚本——什么时候切换账号、每个账号给老师看什么、点哪些按钮、达成什么业务闭环。
>
> 适用版本：含"时间机器 + 因果干预""经济决策""LLM 接地""演示一键重置"等全部能力的当前主干。

---

## 0. 演示前准备（开场前 2 分钟）

1. 启动后端：在 `backend/` 运行 `uvicorn app.main:app --reload`（默认 `127.0.0.1:8000`）。
2. 启动前端：在 `frontend/` 运行 `npm run dev`，浏览器打开提示的地址（默认 `5173`）。
3. **点一次"重置演示（回到初始）"按钮**（页面顶部"运营沙盘 · 时间机器"右侧红色按钮）。
   - 作用：清空所有工单与预算、把时间机器复位、并自动预置 3 条历史已关闭样例工单。
   - 这样无论之前演练到哪一步，按一下即回到干净的初始状态，老师看到的画面始终一致。

## 1. 总览与"会动的系统"（管理员，约 2 分钟）

**账号**：登录 `admin / admin123`。

**操作与话术**：
1. 先看顶部总览卡片与图表（数据记录、平均 COP、异常数、楼宇对比、COP 排名）。
   - 话术："这是 4–6 月的楼宇能耗数据，系统能识别同建筑基线下的电耗异常和 COP 偏低。"
2. 指向顶部"运营沙盘 · 时间机器"，点击 **启动沙盘（定位到 5 月）**。
   - 话术："系统把'今天'定位到 2026-05-01。指针之前是已发生历史，之后是'未来'。我可以一天天推进时间，**我现在的处置会改变系统的未来**——这就是我们和普通监控大屏最大的不同。"
3. 点 **▶ 下一天** 一两次，让老师看到日期推进、数据边界随之变化、并可能冒出新的异常。
   - 话术："未来会持续按计划产生新异常，所以演示不会出现'修完就没事可做'的死局。"

**这一段证明的逻辑**：系统不是静态演示，而是一个由"模拟时钟 + 因果干预"驱动的、可被操作影响的动态闭环。

---

## 2. 经济决策视角（管理员，约 2 分钟）

**操作与话术**：
1. 进入"经济决策 / 决策"面板，展示：
   - **工单优先级**：按风险、预计损失、SLA、碳排、重复异常综合打分排序。
     - 话术："即使现在还没建任何工单，面板也会直接把**当前实时异常**作为候选（标注'尚未建单'）排好序，告诉我先修哪台最划算。"
   - **今日资源约束派单建议**：在工人数量受限时，给出当天最优派单组合与覆盖的预计损失。
2. （可选）展示预算影响 / ROI 候选：把节能改造从"拍脑袋"变成"看回报"。

**这一段证明的逻辑**：从"发现异常"升级到"用经济指标决定先做什么、给谁做"。

---

## 3. 异常处置时机对照（管理员，约 1.5 分钟）

**操作与话术**：
1. 在异常/工单生成流程里**选择一条异常**，下方会自动出现"异常处置三情景对照"。
2. 展示三条曲线：**不处理 / 立即处理 / 延迟 3 天处理** 在未来 7 天的影响差异，以及系统给出的决策结论句。
   - 话术："系统用反事实模拟量化了'拖延的代价'，把'要不要现在修'变成可以摆给领导看的数字。"

**这一段证明的逻辑**：决策不仅有排序，还有"时机"的量化论证。

---

## 4. 工单闭环 + 因果干预（核心，约 4 分钟）

这是整场最重要的闭环，**建议完整演一遍**。

**4.1 管理员建单/派单**
1. 用 **一键生成工单队列**（auto-confirm-queue）把当前异常批量转成待处理工单，或对某条异常单独"生成工单"。
2. 说明工单状态机：待确认 → 已派单 → 处理中 → 待复核 → 已关闭（可驳回/忽略），每一步都有时间线留痕。

**4.2 切换到工人，处理工单**
1. 右上角 **退出登录**，改登 `worker_ahu / worker123`。
   - 话术："工人只能看到派给自己的工单和智能问答，看不到管理端的决策/预算页面——这是基于角色的权限隔离。"
2. 工人在"我的工单"里 **接单 → 处理 → 提交复核**，填写实际原因、处置说明等。
   - 指出"相似历史案例"：因为初始预置了历史已关闭样例工单，工人能看到同类设备过去是怎么修的，辅助决策。
3. 指出"修复后能耗/COP"标注的是 **预计值（估算）**，不会冒充实测，保证诚实。

**4.3 切回管理员，复核关闭 + 看因果效果**
1. 退出登录，重新登 `admin / admin123`，在工单里 **复核通过 → 关闭**。
2. 回到"时间机器"，点 **▶ 下一天 / ⏭ 下一周**。
   - 话术："注意这台刚关闭的设备，从处置日往后**不再出现异常**——因为关闭工单触发了对该设备未来数据的因果干预；而**没有处理的设备，异常会继续**。这就是'我的操作改变系统未来'。"

**这一段证明的逻辑**：管理员↔工人的完整协作闭环 + 操作对未来的因果影响，是整个系统的灵魂。

---

## 5. LLM 智能助手（接地，约 1.5 分钟）

**操作与话术**：
1. 在"智能问答"里提一个业务问题，例如："现在有哪些高优先级工单？建议先处理哪台设备？"
   - 话术："助手的回答**接地到真实业务数据**（真实工单/异常），并会校验它引用的实体 ID，避免编造；问知识库类问题时也不会被强行套上工单。"
2. 可对比问一个知识库/规范类问题，展示它不会乱引用工单。

**这一段证明的逻辑**：LLM 不是花瓶，而是被业务数据"约束"过的可信助手。

---

## 6. 权限隔离佐证（可选，约 1 分钟）

如果老师追问权限，可现场佐证：
- 工人账号看不到决策/预算/ROI 等管理端页面；
- 工人不能操作别人的工单、不能自我复核（后端返回 403/409）；
- 前端登录后会自动携带当前用户身份（`operator_id`）访问管理端接口，后端按角色校验。

---

## 7. 收尾

1. 一句话总结："我们把一个能耗监控大屏，升级成了**异常→决策→派单→处置→因果验证**的运维闭环，并用经济指标和反事实模拟支撑决策。"
2. 如需再演一遍：点一次 **重置演示（回到初始）**，整套流程可即时重来。
