## 1. Experimental Purpose and Principles

### 1.1 Experimental Purpose

本实验的目的在于理解软件成本估算的基本思想，并将实验二得到的软件规模度量结果进一步转化为项目工作量、日历工期和人民币口径的软件开发成本。结合课程项目多次 Git 迭代后的封版状态，本报告重新完成以下工作：

1. 根据实验二更新版的软件规模结果，重新估算当前系统的开发工作量和开发成本。
2. 使用软件行业基准数据建立“规模、生产率、工作量、人月、成本”的估算链路。
3. 使用功能点单价法和 NESMA 保守规模口径进行交叉验证。
4. 分析不同成本估算方法结果差异的原因，并说明差异超过 20% 时的处理方法。

### 1.2 Experimental Principles

软件成本估算通常以软件规模度量为基础。实验二更新版已经对当前封版系统重新进行了功能点分析，得到如下结果：

- 主方法：IFPUG 高层级功能点分析法。
- 未调整功能点：UFP = 254。
- 值调整因子：VAF = 1.13。
- 调整后功能点：FP = 254 x 1.13 = 287.02，约为 287 FP。
- 验证方法：NESMA 估算法。
- 验证规模：250 FP。

实验三在此基础上采用基于功能点的软件开发成本估算方法。基本公式如下：

```text
开发工作量（人时） = 软件规模（FP） x 开发生产率（人时/FP） x 调整因子
开发工作量（人月） = 开发工作量（人时） / 每人月工作小时数
软件开发成本 = 开发工作量（人月） x 基准人月费率 + 直接非人力成本
```

本实验参考《2025 年中国软件行业基准数据》（CSBMK-202510）中的公开基准数据，并结合实验三指导书给出的工作日折算口径。主要假设如下：

| 参数 | 取值 | 说明 |
| --- | ---: | --- |
| 开发生产率 | 6.72 人时/FP | 2025 年中国软件行业基准数据，全行业 P50 |
| 每人月工作小时数 | 180 人时/人月 | 实验指导书口径：22.5 天 x 8 小时 |
| 上海软件开发人月费率 | 31,309 元/人月 | 课程项目完成地点按上海口径估算 |
| 北京软件开发人月费率 | 32,198 元/人月 | 用于从北京功能点单价折算上海单价 |
| 北京软件开发功能点单价 | 1,243.52 元/FP | 2025 年中国软件行业基准数据 |
| 工作量调整因子 | 1.00 | 课程实验中暂不额外引入质量、风险或外包调整 |
| 直接非人力成本 | 0 元 | 课程原型暂不计入云资源、商用模型 API、数据采购和差旅费用 |

其中，人月折算如下：

```text
1 人月 = 22.5 天 x 8 小时/天 = 180 人时
```

本项目如果进入真实生产部署，应另行估算云服务器、数据库托管、商业大模型调用、监控告警、数据采购和现场实施等直接非人力成本。本报告为了保持课程实验口径，仅估算软件开发人力成本。

## 2. Experimental Results: Cost of the Course Project

### 2.1 Course Project Overview

本次成本估算对象为本组课程项目“基于大模型的建筑能源智能管理与运维优化系统”。与旧实验三报告相比，当前项目已经从早期“数据查询 + 统计分析 + 智能问答”原型，迭代为面向建筑能源运营闭环的 Web 系统。当前封版范围如下：

- 后端：FastAPI + Pandas，提供 REST API、统计分析、异常解释、CSV 导出、工单、预算、ROI、决策报告和可选 MySQL 持久化能力。
- MCP：基于同一业务服务层暴露 MCP Server，支持数据查询、统计分析、业务闭环、运营报告、知识库检索和智能问答。
- 前端：Vue 3 + Vite，形成“角色登录 / 总览 / 数据浏览 / 统计分析 / 工单中心 / 我的工单 / 决策报告 / AI 助手”工作台。
- 数据：样例能耗数据文件当前包含 4,864 条记录，覆盖 4 栋建筑和 2026-01-01 至 2026-06-01 的演示周期，并包含楼层、区域、设备、环境、人员密度、暖通能耗和制冷负荷等字段。
- 知识库：包含异常诊断、设备维护、建筑类型、术语规则、问答素材和 RAG 检索卡片。
- 业务闭环：支持异常确认、派单、接单、提交处理、复核关闭、忽略归档、时间沙盘、预算 KPI、节能模拟、ROI 改造分析和运营日报。
- 测试：后端测试目录当前包含 19 个测试文件，使用项目虚拟环境可收集 119 项自动化测试。

项目仍属于新开发项目，而不是对外部既有系统的增强维护项目。因此，本实验按新开发软件进行成本估算。

### 2.3 Main Estimation: Productivity and Person-Month Method

主估算方法采用功能点规模、软件开发生产率和上海地区软件开发人月费率进行估算。

1. 计算开发工作量：

```text
开发工作量 = 287 FP x 6.72 人时/FP x 1.00
         = 1,928.64 人时
```

2. 折算为人月：

```text
开发工作量 = 1,928.64 / 180
         = 10.7147 人月
         ≈ 10.71 人月
```

3. 计算软件开发成本：

```text
软件开发成本 = 10.7147 人月 x 31,309 元/人月 + 0
             = 335,465.50 元
             ≈ 33.55 万元
```

因此，按主估算方法，本课程项目当前封版范围的软件开发成本估算结果为：

```text
约 335,465.50 元，即约 33.55 万元人民币。
```

### 2.4 Calendar Time Conversion

若按 4 人团队全职投入估算，则日历工期为：

```text
全职日历工期 = 10.7147 人月 / 4 人
             = 2.6787 月
             ≈ 2.68 月
```

按实验指导书 22.5 个工作日/月折算，约为：

```text
2.6787 月 x 22.5 天/月 = 60.27 个工作日
```

由于本项目为课程项目，实际开发不是全职投入。若按每人约 50% 的课余投入折算，则日历工期约为：

```text
课程实际日历工期 = 10.7147 / (4 x 0.5)
                = 5.3573 月
                ≈ 5.36 月
```

该结果与项目从需求理解、第一次整合、第二次增强、业务闭环补齐、最终文档生成到实验报告更新的持续迭代过程基本一致。

### 2.5 Verification Method 1: Function Point Unit Price Method

为了验证主估算结果，本报告采用功能点单价法进行交叉验证。基准数据中给出北京地区软件开发功能点单价为 1,243.52 元/FP，并说明其他地区可按照当地与北京软件开发人月费率的对应关系折算。上海地区功能点单价估算如下：

```text
上海功能点单价 = 1,243.52 x 31,309 / 32,198
              = 1,209.19 元/FP
```

则课程项目成本为：

```text
软件开发成本 = 287 FP x 1,209.19 元/FP
             = 347,036.35 元
             ≈ 34.70 万元
```

主估算结果与功能点单价法结果对比如下：

| 方法 | 成本结果 | 与主估算差异 |
| --- | ---: | ---: |
| 生产率 + 人月费率法 | 335,465.50 元 | -- |
| 功能点单价法 | 347,036.35 元 | 约 3.45% |

两种方法的差异小于 20%，说明主估算结果具有较好的合理性。

### 2.6 Verification Method 2: NESMA Size Sensitivity Check

实验二更新版中使用 NESMA 估算法得到的验证规模为 250 FP。若使用相同的生产率和上海人月费率进行估算：

```text
开发工作量 = 250 FP x 6.72 人时/FP = 1,680.00 人时
人月 = 1,680.00 / 180 = 9.3333 人月
软件开发成本 = 9.3333 x 31,309 = 292,217.33 元
```

即 NESMA 口径下的软件开发成本约为 29.22 万元。该结果与 IFPUG 主估算结果相比差异约为：

```text
(335,465.50 - 292,217.33) / 335,465.50 = 12.89%
```

差异仍小于 20%，处于可接受范围内。差异主要来自两个方面：一是 NESMA 验证口径不使用 VAF 调整；二是 NESMA 对控制型输入的识别更保守。

### 2.7 Final Cost Conclusion

综合主估算和两种验证结果，本课程项目当前封版范围的软件开发成本可给出如下结论：

| 估算口径 | 规模 | 成本 |
| --- | ---: | ---: |
| IFPUG + 生产率 + 上海人月费率 | 287 FP | 33.55 万元 |
| IFPUG + 上海功能点单价 | 287 FP | 34.70 万元 |
| NESMA + 生产率 + 上海人月费率 | 250 FP | 29.22 万元 |

本报告采用第一种方法作为主结果。因此，本课程项目的软件开发成本估算为：

```text
约 33.55 万元人民币。
```

考虑到功能点单价法给出的结果为 34.70 万元，NESMA 保守规模口径给出的结果为 29.22 万元，本项目合理成本区间可理解为约 29.22 万元至 34.70 万元。

### 2.8 Discussion: What If the Difference Is Greater Than 20%?

如果两种软件成本估算方法得到的结果差异超过 20%，不能简单取平均值，而应当回到估算前提进行复核。本项目可采取以下处理方法：

1. 重新检查系统边界，确认是否将测试脚本、启动脚本、打包脚本、课程文档等非用户可识别功能错误计入。
2. 重新检查 REST API、MCP Tools 和前端页面之间的关系，避免同一业务功能因为多入口暴露而被重复计数。
3. 重新检查功能点分类，尤其是工单、预算、ROI、知识库、智能问答、时间沙盘和 MySQL 持久化能力应归入 ILF、EIF、EI、EO 还是 EQ。
4. 重新选择生产率基准，判断是否应使用全行业 P50、能源管理类项目基准、组织历史数据，还是 P25/P75 等更保守或更乐观的分位数。
5. 检查直接非人力成本是否遗漏，例如云服务、商业大模型接口、数据库托管、数据采购、部署环境和现场实施费用。
6. 结合 Git 提交记录、任务分工、测试数量和实际工时记录进行校准，使成本估算更贴近真实项目环境。

## 3. Your Learning Experiences in This Experiment

通过本次更新，我对软件成本估算有了更完整的认识。实验二的软件规模度量回答的是“系统现在有多大”，而实验三进一步要求把规模转换为工作量、工期和成本。这个过程让我意识到，软件工程管理中的估算不是单一公式，而是一条从需求边界、功能点规模、生产率基准到地区人月费率的连续推理链。

本次报告最大的变化来自项目范围变化。旧实验三报告基于早期原型，使用 105 FP 作为主规模，并得到约 12.27 万元的开发成本。当前项目已经增加角色登录、工单闭环、预算 KPI、ROI 改造分析、决策报告、MCP Server、时间沙盘、可选 MySQL 持久化和更完整的知识库能力，实验二更新版的主规模变为 287 FP。因此，主成本结果也从约 12.27 万元上升到约 33.55 万元。这个变化不是成本公式突然变复杂，而是系统边界和用户可识别功能真实扩大后的结果。

在实验过程中，我体会最深的是成本估算对前提假设非常敏感。例如，本项目使用 IFPUG 更新结果得到 287 FP，而使用 NESMA 保守口径得到 250 FP，最终成本分别约为 33.55 万元和 29.22 万元，差异约 4.32 万元。虽然差异比例仍在可接受范围内，但已经足以影响课程项目的资源评估和排期判断。因此，估算报告必须清楚说明规模来源、生产率来源、人月折算口径和是否计入非人力成本，而不能只给出一个看似精确的数字。

我也认识到公开行业基准数据的重要性。仅凭个人经验很难判断“每个功能点需要多少人时”或“一个人月应该按多少钱计算”，而行业基准数据为生产率、人月费率和功能点单价提供了可追溯依据。对于课程项目而言，使用这些基准数据可以使成本估算更接近真实工程实践。

结合我们自己的课程项目，我进一步理解了软件成本与技术结构之间的关系。本系统虽然是课程原型，但已经包含前端、后端、数据、知识库、MCP、测试和最终交付文档，其成本并不只来自编码。需求分析、接口契约、数据字典、知识库整理、测试验证、多轮集成和业务闭环设计都消耗了实际工作量。实验三让我更加重视这些“没有直接写在界面按钮上，但真实支撑系统交付”的工程活动。

最后，本次实验让我认识到成本估算的价值不是追求绝对准确，而是为项目计划、资源分配和经济评价提供理性依据。对于未来的软件项目，我会尽量在需求阶段就进行功能点识别和成本估算，并在项目迭代过程中持续用实际工作量、功能变更和验收结果修正估算模型。
