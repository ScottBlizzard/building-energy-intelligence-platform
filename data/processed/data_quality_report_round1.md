## 3. 当前样例数据场景覆盖分析

### 3.1 已覆盖的场景

| 场景 | 覆盖情况 |
|---|---|
| 多建筑类型对比 | 教学楼 / 办公楼 / 图书馆 / 实验楼均有覆盖 |
| 连续时序记录 | 152 天 × 8 个时间槽/天 × 4 栋建筑，连续无断层 |
| 日内高峰 / 低谷变化 | 白天（09:00-15:00）电耗明显高于夜间（00:00-03:00） |
| 工作日 / 周末差异 | 教学楼、办公楼在周末负荷和人员密度下降，图书馆保持较高开放度 |
| 冬季到初夏季节变化 | 1～2 月供暖负荷偏高，4 月为肩季低谷，5～6 月制冷负荷回升 |
| 设备异常标记 | 350 条 abnormal 记录，占 7.2%，分散在 4 栋建筑 |
| 分析异常标记 | 351 条 is_anomaly=true 记录，按设备状态、电耗时段阈值和 COP 告警综合生成 |
| COP 计算支撑 | hvac_kwh + cooling_load_kwh 均完整，可直接计算（均值 2.99） |
| 电耗 + 水耗同步记录 | water_m3 字段完整 |
| 温度 / 湿度环境数据 | 环境温度和湿度均有记录，可支撑天气与能耗关系分析 |

### 3.2 尚未覆盖 / 覆盖偏弱的场景

| 场景 | 当前情况 | 说明 |
|---|---|---|
| 节假日行为差异 | 中 | 已体现周末行为差异，但未单独增加节假日标记字段 |

## 4. 异常数据支撑演示分析

### 4.1 异常状态记录分布

| 建筑 | 异常状态记录数 | 占该楼总记录 |
|---|---:|---:|
| 综合教学楼A（BLD-A） | 81 条 | 6.7% |
| 行政办公楼B（BLD-B） | 84 条 | 6.9% |
| 图书信息楼C（BLD-C） | 89 条 | 7.3% |
| 科研实验楼D（BLD-D） | 96 条 | 7.9% |

### 4.2 当前异常判定口径

后端分析服务统一使用以下规则：

```text
is_anomaly = equipment_status != normal
          OR electricity_kwh > 同建筑同小时段均值 + 2σ
          OR average_cop < 2.2
```

其中夜间高负荷是“电耗超过时段阈值且发生在夜间”的异常原因标签，不再作为更宽松的独立规则。当前样例最低 COP 为 2.31，因此主数据集中没有仅由 `COP < 2.2` 触发的记录；COP 仍参与风险评分和诊断解释。

### 4.3 典型异常记录分析

**案例 1：R00001 — 综合教学楼 A，2026-01-01 00:00**
- electricity_kwh=306.37，hvac_kwh=92.19，cooling_load_kwh=236.71
- COP≈2.57，深夜 00:00，人员密度=2.1，equipment_status=abnormal
- **演示价值：** 凌晨低人员密度下出现设备异常，适合演示“非工作时段异常排查”。

**案例 2：R00024 — 图书信息楼 C，2026-01-01 21:00**
- electricity_kwh=402.01，hvac_kwh=117.17，cooling_load_kwh=273.98
- COP≈2.34，高于 2.2 告警线但偏低，equipment_status=abnormal
- **演示价值：** 适合演示“COP 作为诊断线索 + 设备状态异常”的综合解释，而不是低 COP 直接告警。

**案例 3：R04606 — 科研实验楼 D，2026-05-24 15:00**
- electricity_kwh=690.04（全库最高），COP≈2.48，equipment_status=abnormal
- **演示价值：** 初夏高温下午高峰叠加设备异常，适合演示“高负荷超阈值 + 工单优先级”。

### 4.4 异常数据支撑演示的总体评估

当前数据可以支撑：
- 按建筑、楼层和设备筛选异常列表
- 异常原因解释与风险评分
- 时间趋势中标记异常点
- 从异常记录生成工单并跟踪处理状态
- 工单关闭后通过沙盘干预影响未来数据

仍可继续增强：
- 主 CSV 尚未增加独立 `anomaly_type`、`severity` 字段，异常类型由后端分析服务衍生
- 传感器缺失、断电、设备离线等异常尚未进入主 CSV
- 若后续时间允许，可增加更多真实设备台账字段，减少派生逻辑

## 5. 与外部数据源口径对比

| 维度 | 外部真实数据口径 | 当前样例 | 差距说明 |
|---|---|---|---|
| BDG2 时间粒度 | 小时级（hourly） | 3 小时间隔 | 当前粒度偏粗，适合课程演示 |
| BDG2 计量类型 | electricity/heating/cooling/steam 等多种 | electricity + hvac + cooling_load | 当前为演示简化版 |
| BDG2 建筑数量 | 大规模多建筑 | 4 栋 | 课程项目适量 |
| SHIFDR HVAC 细节 | 子系统级功率、需求响应事件 | hvac_kwh 单字段 | 未覆盖真实调控事件 |
| NYC LL84 粒度 | 年度披露 | 3 小时动态 | 口径不同，互补 |
| NYC LL84 水耗字段 | energy + water benchmarking | water_m3 已有 | 字段设计有真实政策背景依据 |

## 6. 后续优化建议

1. 增加 `anomaly_type`、`severity` 和 `recommended_action` 字段，减少前后端推断逻辑。
2. 补充传感器缺值 / 离线异常，支持更丰富的知识库问答场景。
3. 补充真实建筑面积字段，增强 EUI（能耗强度）指标计算可信度。
4. 增加节假日标记字段 `is_holiday`，区分工作日和节假日行为差异。
5. 拆分 electricity_kwh 为照明、插座、实验设备等 end-use 子项，与 NREL EULP 口径对齐。

## 7. 本报告总结

- 当前样例数据可直接用于接口联调、演示和最终项目验收。
- 字段设计覆盖核心演示场景，BDG2 / SHIFDR / NYC LL84 的设计思路均有对应。
- 数据规模为 4,864 条，覆盖 2026-01-01 至 2026-06-01，已经超过课程要求的 1,000 条。
- COP 告警阈值统一为 2.2；当前数据中 COP 最低 2.31，无低 COP 直接触发记录。
- 当前主 CSV 保持字段简洁，异常类型和处置建议由后端分析服务衍生生成。
