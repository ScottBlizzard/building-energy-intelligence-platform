# 四人分工与个人替换说明

这份说明的目标很简单：让四个人先协作完成一套统一测量结果，再各自生成自己的报告版本。

## 1. 哪些部分可以基本一致

下面这些内容可以四个人保持大体一致：

- 实验背景
- 项目简介
- 系统边界与测量范围
- 采用的测量方法
- FP 统计表和最终结果
- 主要参考文档

原因很直接：这次实验的测量对象本来就是同一个课程项目，小组的最终测量结论应当一致。

## 2. 哪些部分建议每个人自己改

下面这些内容不要四个人完全一样：

- 学号、姓名、封面信息
- `个人贡献说明`
- `实验学习体会`
- 如果你们想更稳：在“实验步骤”里加 1 句自己具体参与了什么

## 3. 推荐分工

### 许奕

- 负责组织实验流程和统一口径
- 先把 [02-FP测量工作表.md](/d:/软件经济管理/实验二/02-FP测量工作表.md) 的边界与范围敲定
- 汇总每个人给出的候选功能和分类意见
- 把最终结果回填到 [01-实验报告模板.md](/d:/软件经济管理/实验二/01-实验报告模板.md)

### 王天一

- 重点看系统结构、接口和业务流程
- 负责给出 `EI / EO / EQ` 的第一版识别结果
- 重点判断哪些功能只是实现细节，哪些才是真正的用户可识别事务功能
- 对“总览”“分析结果”“智能问答”这类边界模糊项给出分类理由

### 马源胜

- 负责给出 `ILF / EIF` 的第一版识别结果

### 周由

- 帮助确认某个候选功能是否真的“被用户识别为一个完整功能”
- 检查查询、分析、图表、问答这些功能在页面上是如何呈现的
- 协助把报告排版成更适合提交的格式

## 4. 每个人最后至少要改哪里

每个人在复制模板后，至少完成下面 4 件事：

1. 把自己的学号、姓名和文件名改掉
2. 在附录里的 `个人贡献说明` 写成自己的版本
3. 把 `实验学习体会` 改成自己的体会
4. 检查文中是否还留有 `【待替换】`、`【个人填写】`

## 5. 很实用的建议

- 不要一开始就争论公式，先把“系统边界”和“候选功能清单”定下来。
- 不要机械地把每个 API 都算一个功能点组件，FP 看的是用户可识别功能。
- 四个人最后的最终 FP 结果必须一致，不一致会显得小组根本没统一口径。
- 体会部分尽量写你自己在实验里遇到的难点，这样最自然，也最不容易撞车。
