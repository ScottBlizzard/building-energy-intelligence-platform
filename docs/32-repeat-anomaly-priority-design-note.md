# 32 · 重复异常与派单优先级的设计说明与已知盲区

> 记录时间：2026-06-12
> 来源：对 `decision_service.py` 的代码走读与设计审查。
> 关联：`docs/29`（数据质量验收规范）、`docs/28`（ROI 方法学）、`docs/30`（系统参考底稿）。

---

## 0. 问题背景

系统的派单优先级评分（`decision_score`）中包含一个 `repeat_component` 分量，用于体现"反复异常设备"的慢性问题权重。

但在审查实现细节后，发现该机制存在一个**已知设计盲区**，本文对此完整记录，并说明补救机制与答辩口径。

---

## 1. 派单优先级评分结构

`rank_open_work_orders()` 为每一条待派单计算综合决策分 `decision_score`（满分 100）：

```
decision_score = risk_component    (满分 40)   # 风险分归一
              + loss_component     (满分 25)   # 损失金额归一
              + sla_component      (满分 15)   # SLA紧迫度
              + carbon_component   (满分 10)   # 碳排归一
              + repeat_component   (满分 10)   # 重复异常次数
```

其中 `repeat_component` 的计算：

```python
repeat_count = repeat_counts.get(equipment_id, 1)
repeat_component = min(10.0, max(0, repeat_count - 1) * 4.0)
```

| 历史异常/工单次数 | repeat_component |
|---|---|
| 1次（首次） | 0 分 |
| 2次 | 4 分 |
| 3次 | 8 分 |
| ≥ 4次 | 10 分（封顶） |

---

## 2. 核心盲区：`repeat_count` 数的是工单，不是原始异常

```python
# decision_service.py
repeat_counts = _equipment_repeat_counts(list_work_orders() + candidates)

def _equipment_repeat_counts(orders):
    counts = defaultdict(int)
    for order in orders:
        equipment_id = order.get("equipment_id")
        counts[equipment_id] += 1
    return counts
```

**`repeat_count` 统计的是该设备历史上产生过多少张工单（含当前候选），而非原始数据集里该设备有多少条异常记录。**

### 2.1 由此产生的盲区场景

```
1月：设备 AHU-C-3F-02 出现异常 → 管理员未建工单 → repeat_count 不增
2月：同设备再次异常            → 管理员未建工单 → repeat_count 不增
5月（沙盘当前）：同设备再次异常 → 系统计为首次（repeat_count = 1）
                                 → repeat_component = 0 分
```

**结论**：一台已经在过去数月内反复出问题、但从未被正式建单的设备，其"慢性程度"在派单优先级层面是**完全不可见**的。它在今天的优先级评分中，与一台真正的首次故障设备得分相同。

### 2.2 为什么会这样设计

这是一个有意识的工程取舍：

- **工单记录 = 已被管理层"承认"的问题**。只有进入工单流程的异常，才是运维团队明确意识到并决定处置的事件。用工单数量衡量重复性，避免系统"自动放大"那些被人工判断为不重要的轻微异常。
- **原始异常记录噪声较高**。部分异常可能因季节、偶发负荷波动产生，并非设备真实退化信号。用工单作为过滤器，引入了人工确认这一层质量把关。

---

## 3. 补救机制：ROI 候选池直接扫原始异常

系统为"慢性顽疾"设备提供了另一条独立路径：

```python
# decision_service.py
def find_roi_candidates_from_repeated_anomalies(...)
```

该函数**直接扫描原始数据集**，按设备统计全量历史异常次数，不依赖工单记录。反复出现异常的设备会被推入"改造候选池"，在「改造分析」页签中出现，触发 ROI 经济性评价流程。

### 两套机制的分工

| 机制 | 数据来源 | 回答的问题 |
|---|---|---|
| `repeat_component`（派单优先级） | 历史**工单**记录 | 今天这台设备，该排到派单队列的第几位？ |
| ROI 候选池（改造分析） | 原始**异常记录** | 这台长期反复出问题的设备，是否到了值得改造的阈值？ |

这两套机制互补：**短期运维决策用工单历史；中长期改造决策用异常历史**。

---

## 4. 潜在改进方向（不纳入当前版本）

若未来版本希望让派单优先级也能感知"未建单的历史异常"，可以考虑：

1. **混入原始异常计数**：把该设备在可见窗口内的全部历史异常记录数也纳入统计，加权后与工单次数合并。
2. **引入"连续异常天数"字段**：扫描该设备最近 N 天内有多少天出现过异常，连续出现视为比单次更严重的信号。
3. **分开展示两个维度**：在派单卡上同时显示"工单历史次数"与"数据库原始异常次数"，让管理员自行判断。

---

## 5. 答辩口径

**如果评审问："一台设备历史上反复出问题，但没建工单，系统怎么知道它是慢性问题？"**

> 「我们的派单优先级中的 `repeat_component` 基于工单历史统计，这意味着未被建单的历史异常不会在短期派单维度中被感知。这是一个有意识的设计取舍——工单是管理层"承认这是问题"的信号，我们用它过滤噪声。
>
> 但我们同时提供了另一条路径：改造分析的 ROI 候选池直接扫描原始异常记录，能识别那些"长期低频、从未被正式建单"的慢性退化设备，并触发改造经济性评价。两套机制各有侧重：工单历史管短期派单优先级，异常历史管中长期改造决策。」
