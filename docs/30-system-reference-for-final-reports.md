## 0. 一句话定位

一个**以"时间沙盘"驱动因果**、把每条暖通能耗异常**量化为经济影响（电费/碳/风险）**、
在**有限工人产能下做优先级派单与闭环复核**、并把执行结果**反馈到预算考核与改造投资决策**
的建筑能源业务闭环系统。

核心闭环：**监测 → 量化 → 决策 → 派单执行 → 复核闭环 → 反馈（预算/考核/改造）**，
外层由**沙盘时钟**推进，形成"操作影响未来"的因果链。

业务逻辑分两层：
- **分析/决策层（系统的"魂"）**：异常量化、风险评分、反事实情景、预算执行与考核、改造 ROI。回答"该不该做、值不值、谁优先"。
- **执行/协作层（系统的"形"）**：多角色工单、资源约束派单、忙闲锁定、复核关闭、现场照片。回答"谁去做、做完没、反馈到哪"。

### 1.1 数据库设计（MySQL，供 SDS 数据库章节）

系统采用**可切换的双存储后端**，由环境变量 `DATABASE_URL` 决定：

- **未配置**（默认）：业务状态存 JSON、原始读数存 CSV，零依赖、便于离线演示与单测。
- **已配置**：通过 SQLAlchemy ORM 接入 **MySQL**，连接串形如
  `mysql+pymysql://user:password@127.0.0.1:3306/energy?charset=utf8mb4`。

切换不改动任何业务逻辑——`data_loader` 与三个 store 的 6 个读写函数会在运行时判断后端，
分析层始终用 pandas 在内存计算（先 `read_sql` 取数再聚合），**所以查询速度不退化**，
MySQL 的价值在于持久化、事务（ACID）、并发与标准化表结构（而非加速分析）。

**表结构（4 张）**

| 表 | 主键 | 关键列（建索引） | 完整对象 | 说明 |
| --- | --- | --- | --- | --- |
| `energy_readings` | 自增/无 | building_id、equipment_id、timestamp | —（按 CSV 列展开） | 原始能耗读数，由 `init_db` 从数据集导入 |
| `work_orders` | work_order_id | status、assignee_id、building_id、source_record_id、pos | `data`(JSON) | 工单：可检索字段单列 + 完整字典存 JSON 列 |
| `budgets` | id(自增) | building_id、year、month | `data`(JSON) | 月度预算 |
| `sim_state` | id(=1) | sim_current_date、sim_start_date | `data`(JSON) | 时间沙盘状态（单行，含 interventions）；列名避开 MySQL 保留字 `current_date` |

> 设计取舍：可检索/统计字段单独建列并加索引以支持 SQL 过滤；其余易变字段（工单时间线、
> 现场照片 base64、预算明细）放进 `JSON` 列，既保持与原对象结构 100% 一致，又避免为几十个
> 字段做僵硬映射。MySQL 5.7+/8 原生支持 JSON 列。

## 3. 业务逻辑详解（核心，供全部四份报告）

### 3.1 时间沙盘（因果引擎）

- 服务端维护"当前日期"指针；所有查询按 `timestamp <= current_date` 切片，未来数据隐藏。
- **推进时间**（advance）：揭开后续数据。
- **定时故障**（保证时间推进始终"有事发生"）：首次故障在起点后 `3` 天，之后每 `3` 天，
  在去重设备序列中每 `2` 台取 1 台分阶段劣化。
- **维修干预**：审批/关闭维修工单 → 对该设备打"已修复"标记 → 抑制其未来异常（修过的不再复发，没修的继续恶化）。
- **时间线自洽**：自动报告生成时间、工单时间、预算结算口径全部跟随沙盘，不用现实时间。

### 3.2 异常检测（情景化基线 + 均值 +2σ）

- **基线**：按 `建筑 × 小时时段` 分组，求该时段电耗均值 `slot_mean` 与标准差 `slot_std`。
- **判异常**：`electricity_kwh > slot_mean + 2 × slot_std`（σ 系数 `_ANOMALY_SIGMA = 2.0`）。
- 辅助标记：`low_cop`（COP < 目标 2.2）、`night_high_load`（22:00–06:00 高负荷）、设备硬故障（status ≠ normal）。
- 意义：避免全局单一阈值，"白天高负荷不算异常、深夜同样负荷才算"。

### 3.3 业务影响量化（把异常翻译成"钱/碳"）

**浪费电量（取最大，避免重复计）：**
```
over_expected = max(0, electricity_kwh − baseline_mean)          # 超时段期望的多耗
cop_waste     = max(0, hvac_kwh − cooling_load / COP_TARGET)     # 仅 low_cop 时计入，物理推导
wasted_kwh    = max(over_expected, cop_waste)                    # 取 MAX，不求和
```
**金额与碳：**
```
wasted_cost_yuan      = wasted_kwh × 0.82            # 电价 0.82 元/kWh
carbon_kg             = wasted_kwh × 0.5703          # 碳因子 0.5703 kgCO₂/kWh（生态环境部 2022）
estimated_saving_yuan = wasted_cost_yuan × 0.65      # 处置后可回收比例（保守，运维难一次根治）
```

### 3.4 风险评分（透明可加 0–100，无凭空底分）

```
risk = status_component + overuse_component + cop_component + night_component
```
| 分量 | 权重 | 触发/归一方式 |
| --- | --- | --- |
| 设备硬故障 status | 30 | status ≠ normal 计满分 |
| 电耗超期望 overuse | 35 | 按"超时段期望均值的相对幅度"归一，≥35% 即占满 |
| COP 低效 | 20 | 按 (2.2 − COP)/2.2 归一 |
| 夜间高负荷 night | 15 | 22:00–06:00 高负荷计满分 |

- **严重度 / SLA**：risk ≥ 70 → 高 / 8h；≥ 45 → 中 / 24h；否则 低 / 72h。
- 设计要点：每个信号只计一次、可逐项拆解（答辩时可现场拆"为什么 78 分"），无重复计分、无基础分。

> 说明：前端某处历史上出现过"78 分 ×40%=31.2"的派生展示口径，已在前序迭代统一为单一 0–100 口径。

### 3.5 反事实情景（决策的经济性论证）

针对一条异常，对未来窗口模拟三种处置：**立即处理 / 延迟 N 天 / 完全不处理**，
分别给出累计损失（kWh/元/碳）与新增异常暴露次数，并算出"立即 vs 延迟""立即 vs 不处理"的可节省量。
- 若三者无显著经济差异（状态/能效类告警），明确提示"无可量化差异、建议巡检"，不展示三张相同的 0 元卡片（`no_economic_difference` 标志）。

### 3.6 资源约束派单（有限人力下的优先级调度）

- **对口分工**：按设备类型路由——空气处理机组→`worker_ahu`；冷水机组/冷却塔→`worker_chiller`；风机盘管→`worker_fcu`。
- **动态容量**：派单容量 = **当前空闲工人数**（非手填）。每轮每个空闲工人先派 1 单，同工种其余工单进入"延后排队"，给出可解释延后原因。
- **忙/闲锁定**：工人持有 `已派单/处理中` 工单 = 忙碌，**不能再被派第二单**（后端 `WorkerBusyError`→HTTP 409 强约束；前端下拉禁选并自动改派空闲工人）。提交复核后转空闲，当天可继续接单。
- **自动待确认队列**：只取最近 7 天内风险最高的前 10 条（匹配 3 工人产能，不灌全量）。

### 3.7 工单状态机 / 权限 / 现场照片

- 生命周期：`待确认 → 已派单 → 处理中 → 待复核 → 关闭 / 驳回`。
- 权限：管理员（创建/派单/复核/忽略），工人（接单/提交）。
- **现场照片**：工人端选择图片 → 前端压缩为 base64（最长边 1024px）随工单持久化；
  管理员/工人点击缩略图 → **居中大图灯箱**（× 按钮 / 点遮罩 / Esc 关闭）。

### 3.8 预算管理（独立基线 + 月末预测 + 闭环修正）

**基线（季节中性、独立于被考核月实测，避免循环论证）：**
```
季节中性日均 = mean( 每日总电耗 / 该日所属月份季节系数 )
期望运行水平 = 季节中性日均 × 当月季节系数 × 30
预算 budget_kwh = 期望运行水平 × 目标系数 0.97
```
季节系数表（1–12 月）：1.06 / 1.02 / 1.01 / 0.94 / 0.97 / 1.04 / 1.15 / 1.18 / 1.02 / 0.95 / 0.98 / 1.08。

**执行与预测：**
```
执行率       = 当月实际电耗 / 预算
月末预测电耗 = 当月实际 / 已观测天数 × 30          # 仅进行中月份有意义
预测执行率   = 月末预测电耗 / 预算
```
- 目标系数 0.97 的含义：**维持现状（不处置异常/不改造）约落到 1/0.97≈103% 而略超标**；处置异常、推进改造能把实际压回目标线内——这就是"超预算"的可解释来源。
- **闭环修正（Closure Impact）**：已关闭工单减少的预测电量/回收金额会修正月末预测执行率（执行得好→预测转健康）。

### 3.9 年度 KPI 考核评分（满分 100，扣分可解释）

> 已结算月份用**实际执行率**计分（不再做 30 天归一——否则 31 天月被低估、28 天月被高估，会出现"实际超预算却给满分"的口径矛盾）。

| 维度 | 权重 | 扣分规则 |
| --- | --- | --- |
| 预算控制（主项） | 40 | 实际执行率每超目标线 1% 扣 4 分，超 10% 即扣满 40 |
| COP 达标 | 15 | COP 达标率 < 80% 时按差距扣分 |
| 异常高发 | 15 | 单月异常 > 10 件按数量扣分 |
| 异常响应及时率 | 10 | 及时率 < 80% 时按差距扣分 |

- **等级**：avg ≥ 85 → A；75–85 → B；60–75 → C；< 60 → D。
- 现状（沙盘起点 1–4 月）：四栋普遍略超目标线 → 整体 **B**；超预算月份得 69–90、未超月份得 100。"整月超预算"拿不到 A，处置异常把执行率压下来才能升 A——与闭环叙事一致。

### 3.10 改造分析 ROI（投资决策，供 SEE 重点）

对反复异常设备做"要不要改造"的工程经济学评价。

**经济指标（核心公式）：**
```
逐年净现金流  CFt = 年净节省 × (1 + 电价递增 3%)^(t-1)
年净节省      = 年电费节省 − 年增量运维             # ≤0 即如实"不经济"，无 0.15 兜底
NPV(8%)       = −投资 + Σ CFt / (1 + 8%)^t
IRR           = 使 NPV = 0 的折现率（二分法）
EAA           = NPV × r / (1 − (1+r)^(−n))          # 等额年值，寿命不同才公平
动态回收期    = 折现累计现金流首次 ≥ 投资 的年份（线性插值）
含碳价 NPV    = 把年减碳 × 碳价 80 元/吨 计入现金流（加分情景，非主结论）
```
**主判据**：先筛 NPV(8%) > 0，再按 **EAA 最大**择优；动态回收期仅作辅助。

**关键参数：**
- 社会折现率 8%（《建设项目经济评价方法与参数（第三版）》）；敏感性取 5% / 8% / 10%。
- 电价递增 3%/年；电价 0.82 元/kWh；碳因子 0.5703 kgCO₂/kWh；碳价 80 元/吨。

**节能率口径**：实测年化电耗 × 措施基准节能率 × 能效差调节（COP 已达标的设备拿不到满额）。

**投资口径（增量法，docs/28 §2.3）**：
- 变频/智能控制/综合方案 = 改造直接投入（本就是增量）；
- 高效设备更换 = 高效型相对标准型的**差价**（到寿命更换只算差价）。

**单台设备基准 / 措施投资（节选）：**

| 设备类型 | 典型功率 | 寿命(年) | 目标COP | 变频 | 高效更换 | 智能 | 综合 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 冷水机组 | 350 kW | 15 | 3.6 | 8.0万 | 12.0万 | 4.5万 | 18.0万 |
| 冷却塔 | 45 kW | 12 | 3.2 | 3.0万 | 4.0万 | 1.8万 | 7.0万 |
| 空气处理机组 | 55 kW | 15 | 3.4 | 4.5万 | 6.0万 | 2.5万 | 11.0万 |
| 风机盘管 | 2.5 kW | 10 | 3.0 | 0.3万 | 0.5万 | 0.25万 | 0.8万 |

措施基准节能率：变频 20% / 高效更换 35% / 智能控制 12% / 综合方案 28%。

## 8. 答辩 Q&A 预案（供讲稿，老师常问）

1. **"这条异常凭什么 78 分？"** → 现场拆：故障30 + 超耗35×归一 + COP20×归一 + 夜间15，逐项相加，每信号一次。
2. **"为什么预算大多超额？"** → 预算 = 期望水平 × 0.97 目标线；不处置异常约落到 103%。处置/改造能压回线内，这是激励而非错误。
3. **"超预算为什么还能高分？（已修）"** → 已结算月用实际执行率计分、预算控制占 40 权重；整月超预算拿不到 A。
4. **"回收期为什么不是 0？"** → 用折现累计现金流插值，增量法投资 + 3% 电价递增，样例 2.7 年，可解释。
5. **"为什么寿命不同的方案用 EAA 比？"** → NPV 偏向长寿命项目，EAA 等额年值才公平；先筛 NPV>0 再按 EAA 择优。
6. **"工人能无限接单吗？"** → 不能，忙碌锁：持有活动工单期间禁派，后端 409 强约束；提交后转空闲。
7. **"数据是真的吗？"** → 合成数据集，但口径经 L0–L3 验收、公式来自国标/暖通经验值，逻辑可追溯、可复现。
