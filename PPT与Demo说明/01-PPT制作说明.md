# PPT 制作说明

本文档给负责制作项目展示 PPT 的同学使用。目标不是把所有文档照搬到 PPT，而是把系统价值、核心功能、工程完成度和软件经济管理相关内容讲清楚，让展示材料和 Demo 录屏形成一套完整证据。

## 1. PPT 总体目标

PPT 建议控制在 14 到 16 页，适合 8 到 12 分钟项目展示。每页只讲一个重点，尽量使用系统截图、架构图、数据图和关键结论，不要堆大段文字。

PPT 应该回答四个问题：

1. 我们做的系统解决什么问题。
2. 系统具体实现了哪些功能。
3. 项目在需求、设计、测试、经济分析和工程管理上是否完整。
4. Demo 中看到的功能和文档中写的内容是否一致。

## 2. 主要参考材料

制作 PPT 前先看这些文件：

1. `README.md`：了解项目定位、运行方式、功能总览和当前完成状态。
2. `期末提交材料/documents/pdf-latex/01-SRS-软件需求规格说明书.pdf`：提炼需求背景、用户角色、功能需求和业务流程。
3. `期末提交材料/documents/pdf-latex/02-SDD-SDS-软件设计说明书.pdf`：提炼系统架构、模块设计、数据流和接口设计。
4. `期末提交材料/documents/pdf-latex/03-SEE-软件经济分析与评价.pdf`：提炼成本、收益、风险和项目价值。
5. `期末提交材料/documents/pdf-latex/04-SEM-软件工程管理说明.pdf`：提炼分工、过程控制、风险管理和质量管理。
6. `期末提交材料/documents/pdf-latex/05-测试与验收报告.pdf`：提炼测试结论、验收结果和质量证据。
7. `PPT与Demo说明/02-Demo录屏说明.md`：确保 PPT 顺序和 Demo 演示顺序一致。

## 3. 推荐 PPT 结构

### 第 1 页：封面

内容：

1. 项目名称：建筑能耗智能分析与异常工单平台。
2. 课程名称：软件经济管理。
3. 小组成员姓名。
4. 展示日期。

建议画面：使用系统首页或 3D 楼宇视图作为背景，不要使用纯文字封面。

### 第 2 页：项目背景与痛点

重点讲清楚：

1. 建筑能耗数据分散，人工排查效率低。
2. 空调、制冷、照明等能耗异常不容易被及时发现。
3. 传统统计表只能看到数据，不能形成“异常发现到处理闭环”。
4. 管理人员需要一个能查询、分析、定位、解释和跟踪处理的平台。

建议配图：用一张“数据采集、异常分析、工单处理、管理决策”的流程图。

### 第 3 页：项目目标与范围

重点讲清楚：

1. 系统面向校园建筑能耗管理场景。
2. 支持建筑、楼层、时间范围的查询与分析。
3. 支持异常识别、异常解释、三维楼层联动、工单处理和决策报告。
4. 项目范围聚焦“自动检测与管理辅助”，不做真实硬件采集和真实支付结算。

建议配图：用范围边界图，区分“已实现”和“未纳入范围”。

### 第 4 页：用户角色与业务闭环

重点讲清楚：

1. 管理者查看总体风险、统计分析和报告。
2. 运维人员处理异常工单。
3. 项目系统把数据浏览、异常分析、楼层定位和工单状态连接起来。

建议配图：业务闭环图。流程可以写成：

数据读取 -> 查询筛选 -> 统计分析 -> 异常判断 -> 异常解释 -> 创建工单 -> 处理完成 -> 状态持久化。

### 第 5 页：系统总体架构

重点讲清楚：

1. 前端负责页面展示、筛选交互、3D 楼宇、图表和工单操作。
2. 后端负责 API、数据读取、统计计算、异常识别、工单持久化和 AI 调用封装。
3. 数据层使用 CSV 和 JSON，适合课程项目部署与演示。
4. 外部大模型通过环境变量配置，真实密钥不进入 GitHub。

建议配图：从 `02-SDD-SDS-软件设计说明书.pdf` 截取或重画系统架构图。

### 第 6 页：数据设计与数据规模

重点讲清楚：

1. 数据覆盖 2026-03-01 到 2026-06-01。
2. 样本数量为 2,976 条。
3. 数据维度包括建筑、楼层、时间、用电、空调电耗、水耗、制冷量、COP、设备状态和异常标签。
4. 数据经过增强，能够支撑趋势分析、楼层分析、异常识别和工单处理。

建议配图：数据字段表或数据维度示意图。

### 第 7 页：核心功能总览

建议列出 6 个功能模块：

1. 数据总览：关键指标、风险概览、3D 楼宇状态。
2. 数据浏览：按建筑、楼层、时间查询原始记录。
3. 统计分析：按筛选范围展示异常明细、趋势、原因和建议。
4. 工单中心：选择异常、分配责任人、处理完成并持久化。
5. 决策报告：输出管理摘要、节能建议和风险提示。
6. AI 助手：基于系统数据进行解释和问答。

建议配图：系统导航栏截图。

### 第 8 页：数据总览与 3D 楼宇

重点讲清楚：

1. 首页能快速看到总体运行状态。
2. 3D 楼宇按楼层展示健康状态。
3. 红色楼层表示存在异常，绿色楼层表示健康。
4. 点击异常楼层可以联动到统计分析页面。

建议截图：截取 3D 楼宇视图，最好包含红色异常楼层和绿色健康楼层。

### 第 9 页：统计分析与异常解释

重点讲清楚：

1. 统计分析可以按建筑、楼层和时间范围筛选。
2. 选定异常楼层后，异常明细优先展示。
3. 异常解释包括触发规则、指标证据、严重程度和处理建议。
4. 如果选择健康楼层，系统只展示健康状态，不展示无意义的异常模块。

建议截图：选择“综合教学楼A · RF 屋顶”等异常楼层后的统计分析页面。

### 第 10 页：数据浏览与导出

重点讲清楚：

1. 数据浏览用于查看全量记录，不和 3D 楼层强联动。
2. 支持建筑、楼层、时间范围查询。
3. 支持 CSV 导出，便于后续统计或报告引用。
4. 数据浏览更偏“原始数据查询”，统计分析更偏“异常诊断”。

建议截图：数据浏览表格和筛选条件。

### 第 11 页：异常工单处理中心

重点讲清楚：

1. 可以从异常记录中创建工单。
2. 创建时选择建筑、楼层、异常和负责人。
3. 工单进入处理中后显示黄色并置顶。
4. 完成后显示绿色并移动到底部。
5. 工单状态写入本地运行数据，刷新后仍能保留。

建议截图：工单创建区域、处理中工单、已完成工单各截一张。

### 第 12 页：决策报告与 AI 辅助

重点讲清楚：

1. 决策报告面向管理者，强调风险摘要和改进建议。
2. AI 助手用于把数据结果转成自然语言解释。
3. 系统支持多个外部模型配置，但真实 API Key 不进入仓库。
4. 即使没有外部密钥，也可以使用系统内置分析结果完成主要演示。

建议截图：决策报告页面和 AI 助手回答页面。

### 第 13 页：测试与验收结果

重点讲清楚：

1. 后端自动化测试已通过。
2. 前端构建已通过。
3. 正式 LaTeX 文档已生成 PDF。
4. 项目已经整理为可提交材料。

可以写：

1. 后端测试：75 项通过。
2. 前端构建：通过。
3. 文档编译：正式 PDF 已生成。
4. 安全检查：真实 `.env` 和 API Key 不进入提交材料。

建议截图：测试通过命令行截图或 `05-测试与验收报告.pdf` 中的测试结论页。

### 第 14 页：软件经济分析

重点讲清楚：

1. 系统通过减少人工巡检和缩短异常处理时间产生管理价值。
2. 异常早发现可以降低能耗浪费和设备维护风险。
3. 课程项目采用轻量部署方案，降低开发和维护成本。
4. 经济评价包括成本估算、收益来源、风险因素和综合评价。

建议配图：成本收益对比表或项目价值雷达图。

### 第 15 页：工程管理与团队协作

重点讲清楚：

1. 项目采用两轮任务分配与两次整合。
2. 后端、数据与 AI、前端与集成、最终整合各有明确边界。
3. 通过接口契约、测试计划、集成清单和正式文档控制质量。
4. 真实密钥、运行环境和最终提交材料做了隔离。

建议配图：团队分工表或过程时间线。

### 第 16 页：总结与后续改进

重点讲清楚：

1. 项目已形成完整系统、完整文档、完整测试和可展示 Demo。
2. 系统实现了从能耗数据到异常工单的管理闭环。
3. 后续可扩展真实传感器接入、数据库部署、权限管理和移动端提醒。

建议画面：用系统首页截图作为收尾。

## 4. 必须准备的截图

PPT 至少准备这些截图：

1. 系统首页和 3D 楼宇总览。
2. 点击异常楼层后的统计分析页面。
3. 异常明细和异常解释区域。
4. 数据浏览页面。
5. 工单创建、处理中、已完成三个状态。
6. 决策报告页面。
7. AI 助手问答页面。
8. 测试通过或文档 PDF 目录页面。

截图要求：

1. 浏览器缩放建议设置为 100% 或 90%。
2. 截图前关闭无关标签页和隐私信息。
3. 不要截取 `.env` 文件、API Key 或私人聊天内容。
4. 每张截图只保留和该页主题相关的区域。

## 5. 风格建议

1. PPT 主色建议参考系统页面的蓝绿色、浅米色和深色文字，保持统一。
2. 每页标题要直接，例如“异常分析如何定位问题”，不要只写“功能展示”。
3. 每页正文控制在 3 到 5 个短句。
4. 图比文字重要，能用系统截图表达的就不要写长段落。
5. 不要把正式报告整页截图塞进 PPT，只提炼结论。
6. 不要出现真实 API Key、个人账号、未整理的命令行报错。

## 6. 和 Demo 同学的衔接

PPT 和 Demo 的顺序建议保持一致：

1. PPT 先讲系统背景和架构。
2. Demo 展示首页、3D 楼宇、统计分析、工单中心、决策报告和 AI 助手。
3. PPT 后半部分补充测试、经济分析和工程管理。

如果时间有限，PPT 中功能部分不要讲太细，把细节留给 Demo。PPT 负责说明“为什么做、做了什么、完成度如何”，Demo 负责证明“系统真的能运行、功能真的能操作”。
