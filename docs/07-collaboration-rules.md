# 协作与集成规则

这份文档的目标是把多人协作的风险前置压住，尽量把大规模冲突留在集成前解决。

## 1. 当前协作模式

项目采用“两轮任务 + 两次整合”的协作方式：

1. 第一轮：各自完成可接入骨架的基础版本
2. 第一次整合：统一字段、接口、页面流程
3. 第二轮：各自补完整功能与修正项
4. 第二次整合：联调、测试、封版提交

## 2. 目录归属建议

- `backend/`：主要由技术负责人负责
- `frontend/`：主要由前端与集成负责人负责
- `data/`：主要由数据与 AI 负责人负责
- `knowledge_base/`：主要由数据与 AI 负责人负责
- `docs/`：主要由项目经理负责统筹，其他成员补自己模块内容
- `scripts/`：由项目经理或技术负责人统一维护

## 3. 不建议随意修改的内容

- `data/dictionaries/energy_records_dictionary.csv`
- `docs/06-api-contract.md`
- 后端接口路径
- 前端页面主导航结构

这些内容如果要改，必须先同步。

## 4. 第一次任务阶段的目标

- 数据字段冻结
- 前后端能联通
- 每个模块都有基础版成果
- 演示主流程能够串起来

## 5. 第一次整合要检查什么

- 字段名是否一致
- 时间格式是否一致
- 前端筛选参数和后端接口是否一致
- 问答模块引用路径是否有效
- 文档是否与代码结构一致

## 6. 第二次任务阶段的目标

- 图表和分析逻辑补全
- 问答质量提升
- 交互体验优化
- 文档和演示脚本收口

## 7. 减少冲突的具体做法

- 每个人尽量只改自己负责目录
- 共同文件优先由你统一改
- 不要同时多人改 `README.md`、主页面和接口契约
- 一切需要跨模块协调的改动，先写到文档里
