## 1. 文档目的

本文档对应课程期末项目要求中的软件工程管理内容，说明项目范围、项目计划、团队分工、过程监控与控制、质量管理、风险管理、配置管理和最终交付管理。

## 2. 项目范围管理

### 2.1 本期范围

本期必须完成的范围：

- 建筑能耗样例数据集与数据字典。
- 后端 REST API：数据查询、统计分析、导出、问答、工单。
- 前端 Web 工作台：总览、数据浏览、统计分析、工单、决策报告、AI 助手。
- 三维楼层风险态势视图。
- 异常解释和异常工单闭环。
- 外部大模型可选接入。
- MCP Server：数据接入、查询统计、异常解释、运营报告和智能问答。
- 自动化测试、启动脚本、验证脚本和最终交付文档。

### 2.2 非本期范围

以下内容作为后续扩展，不作为期末项目必须完成项：

- 真实传感器和楼宇自控系统接入。
- 多用户登录、权限、审计日志和组织管理。
- 企业级数据库迁移和高可用部署。
- 正式机器学习模型训练平台。
- 长周期真实节能效果审计。

明确范围边界的原因是避免项目无限扩张，保证课程周期内完成可运行、可验收、可演示的系统。

## 4. 项目计划

### 4.1 阶段划分

| 阶段 | 目标 | 主要产出 |
| --- | --- | --- |
| 项目初始化 | 搭建仓库骨架，明确需求和分工 | README、docs、backend/frontend/data/knowledge_base 结构 |
| 第一次任务 | 三人分别完成后端、数据 AI、前端基础能力 | `第一次任务/` 任务文件和初始实现 |
| 第一次整合 | 合并三方成果，修补缺口 | `docs/11-first-integration-summary.md` |
| 第二次任务 | 面向完整演示补齐功能 | `第二次任务/` 任务文件 |
| 第二次整合 | 完成主要功能闭环 | `docs/12-second-integration-summary.md` |
| 封版优化 | 增强 3D、工单、统计分析、LLM、MCP 和文档 | 最终系统、验收报告、SRS/SDS/SEE/SEM |
| 最终提交 | 测试、打包、PPT、视频、Canvas 提交 | zip 包、PPT、演示视频 |

### 4.2 计划控制原则

- 先完成可运行主链路，再补充高级展示。
- 优先保证数据、后端、前端三层接口一致。
- 每次整合后必须能运行系统。
- 最终阶段不再分配大任务，集中做质量收敛和文档补齐。

## 5. 软件过程监控与控制

### 5.1 Git 控制

- 使用 GitHub 作为统一仓库。
- 每轮任务完成后由负责人拉取、检查、整合。
- 真实 `.env` 不提交，防止 API Key 泄露。
- 重要节点形成 commit，便于回滚和追踪。

### 5.2 任务控制

项目通过 `第一次任务/` 和 `第二次任务/` 明确任务边界：

- 每个人负责相对独立的目录和模块。
- 提前说明需要阅读的文档、接触的文件和交付物。
- 由整合者统一处理跨模块联调，减少重复冲突。

### 5.3 接口控制

- REST 接口以 `docs/06-api-contract.md` 为契约。
- MCP 接口以 `docs/16-mcp-integration.md` 为说明。
- 前端调用统一封装在 `frontend/src/lib/api.js`。
- 后端服务层复用，避免 REST 和 MCP 逻辑分叉。

### 5.4 验证控制

项目提供 `scripts/check-project.ps1`，用于封版前统一检查：

- Python 语法检查。
- 后端 pytest。
- 前端 Vite build。

当前脚本已修复为失败即退出，避免测试失败但仍显示成功的误判。

## 6. 质量管理

### 6.1 质量目标

- 系统能从空仓库环境按 README 复现。
- 后端测试全部通过。
- 前端构建成功。
- 演示流程无阻断。
- 文档与当前系统功能一致。
- 提交材料不包含真实密钥。

### 6.2 测试策略

| 测试层次 | 内容 | 证据 |
| --- | --- | --- |
| 单元/接口测试 | 数据、分析、导出、问答、工单、MCP | `backend/tests/` |
| 构建测试 | 前端生产构建 | `npm run build` |
| 集成测试 | REST + 前端 + MCP | `scripts/check-project.ps1` |
| 人工验收 | 页面演示路径和浏览器操作 | `docs/15-project-acceptance-report.md` |

当前基线结果：后端 `75 passed`，前端构建通过。

### 6.3 文档质量控制

最终文档采用两层结构：

- 过程文档：`docs/01` 至 `docs/16`，记录需求、方案、接口、任务和验收过程。
- 最终交付件：`docs/17` 至 `docs/21`，对应 SRS、SDS、SEE、SEM 和提交指南。

## 9. 沟通管理

项目沟通采用“文档先行 + 仓库同步”的方式：

- 需求和分工写入任务文件。
- 接口变化写入 API 契约。
- 整合结果写入总结文档。
- 最终提交要求写入验证说明。
- 运行方式写入 README。

这种方式适合课程小组项目，可以减少口头沟通遗漏。

## 10. 交付管理

最终交付应包含：

- 源代码：`backend/`、`frontend/`、`data/`、`knowledge_base/`、`scripts/`。
- 文档：SRS、SDS、SEE、SEM、用户手册、测试计划、验收报告、MCP 集成说明。
- 演示材料：PPT、演示视频或视频链接。
- 提交说明：最终打包结构和运行方法。

打包时必须排除：

- `.env`
- `data/runtime/`
- `frontend/node_modules/`
- `frontend/dist/`
- `.pytest_cache/`
- 个人无关作业文件
