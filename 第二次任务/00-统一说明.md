# 第二次任务统一说明

这次第二次任务的原则，和第一次任务一样：

**三个人各做各的一块，互相不要交叉写文件，最后由许奕统一整合。**

但第二轮和第一轮有一个很重要的区别：

第一轮的目标是把三块内容分别做成“独立成果包”；  
第二轮的目标是基于**第一次整合后的统一仓库**，把三块内容分别推进到更接近可演示成品的状态。

## 先说清楚第一轮完成质量

### 王天一

- 第一轮完成质量：**高**
- 优点：边界最干净，基本只动了 `backend/`，后端接口稳定性、导出接口和测试都交得比较完整
- 许奕 / Codex 后续补了什么：
  - 把问答接口从纯占位升级成了“规则回答 + 第一轮知识文件引用”
  - 补强了导出文件名规则
  - 把后端测试从 `29` 项补到了 `33` 项
  - 把接口与文档口径重新同步

### 马源胜

- 第一轮完成质量：**高，但边界略有瑕疵**
- 优点：知识库内容质量好，数据检查说明、FAQ、异常诊断和设备维护手册都已经能直接用
- 问题：有一次轻微越界，碰了 `.gitignore`
- 许奕 / Codex 后续补了什么：
  - 把知识库入口重新整理清楚
  - 把这些知识文件真正接入到了问答引用逻辑里
  - 把知识与数据素材在 README 和技术文档里正式挂上入口

### 周由

- 第一轮完成质量：**中上，结构不错，但有“说明大于实现”的问题**
- 优点：组件拆分方向是对的，前端结构比原来清楚很多
- 问题：
  - 一度把没真正实现的能力写得太靠前
  - 有根目录越界文件
- 许奕 / Codex 后续补了什么：
  - 把建筑筛选 + 时间范围筛选 + CSV 导出补成真实功能
  - 补齐了错误状态、导出状态和继续追问交互
  - 删除了不该留在根目录的 `package-lock.json`
  - 把前端说明文档改成和真实实现一致

## 第二轮的正确目标

第二轮不是“大家再各写一遍第一轮”，而是：

1. 王天一继续把 `backend/` 做成更强的独立后端包
2. 马源胜继续把 `knowledge_base/` 和 `data/processed/` 做成更强的独立素材包
3. 周由继续把 `frontend/` 做成更像正式演示系统的独立前端包
4. 最后由许奕统一做第二次整合

### 周由

原则上不要写：

- `backend/`
- `knowledge_base/`

## 第二轮默认冻结的公共基础

这轮仍然不要随便动这些：

- [`data/dictionaries/energy_records_dictionary.csv`](../data/dictionaries/energy_records_dictionary.csv)
- [`data/samples/energy_records.csv`](../data/samples/energy_records.csv)
- [`docs/06-api-contract.md`](../docs/06-api-contract.md)
- 已经存在的后端接口路径和返回字段
- 前端现有四区结构：
  - 总览
  - 数据浏览
  - 统计分析
  - 智能问答

注意：

- 王天一第二轮**可以新增后端接口**，但不要改坏旧接口。
- 周由第二轮默认只使用**当前已经稳定存在的接口**做前端增强，不要把自己的任务建立在“等王天一先新增一个接口”上。
- 马源胜第二轮继续只交资料和结构化素材，不要自己改主样例 CSV 和数据字典。

## 第二轮最合理的方向

基于第一次整合总结，这轮最值得做的是：

1. 前端把图表和展示体验做成更正式的版本
2. 后端把深度分析和问答后端能力再推进一层
3. 知识库与数据素材把“可引用”推进到“更适合检索、更适合评测”的状态

每个人交回来的东西，都应该能被理解为一个**第二轮独立成果包**：

- 王天一：后端增强包
- 马源胜：知识与数据增强素材包
