# 给我自己 / 给 Codex：第二次整合恢复说明

这份文件是写给“未来重新接手这个仓库的我”的。

如果对话上下文突然没了，或者我被重新拉进这个项目，我不要先凭记忆乱判断，而是按这份说明重新熟悉仓库、重新判断三个人第二轮的完成情况、再做第二次整合。

## 先记住这次第二轮协作的真正目标

第二轮不是重新做第一次整合，而是：

- 王天一交一个独立的 `backend/` 增强包
- 马源胜交一个独立的 `knowledge_base/` 和 `data/processed/` 增强素材包
- 周由交一个独立的 `frontend/` 展示增强包
- 最后由我统一做第二次整合

所以第二次整合的目标不是“最终版彻底做完”，而是让这个仓库从“第一次整合可继续开发”推进到“更接近正式演示系统”的状态。

## 我心里的理想样子应该是什么

第二次整合完成后，理想状态大概是下面这样。

### 后端理想状态

- 旧接口路径和返回结构基本不动
- 至少当前旧能力不回归：
  - `overview`
  - `records`
  - `time-summary`
  - `building-comparison`
  - `cop-ranking`
  - `anomalies`
  - `anomaly-reasons`
  - `export/csv`
  - `assistant/query`
- 如果王天一补了检索服务或新分析接口，它们应该是新增能力，不是破坏旧能力
- 后端测试数量和覆盖不应倒退，最好比 `33` 项更多

### 前端理想状态

- 仍然围绕四个主区域：
  - 总览
  - 数据浏览
  - 统计分析
  - 智能问答
- 统计分析至少有一到三块正式图表，不再只是轻量占位
- 问答区展示更像正式产品
- 前端第二轮增强尽量全部基于现有稳定接口完成

### 知识与数据素材理想状态

- `knowledge_base/` 不只是“能引用”，而是更适合检索和评测
- `data/processed/` 不只是第一轮检查说明，而是已经有：
  - 下一轮数据增强规划
  - 更结构化的异常场景目录
  - 更结构化的问答评测集

## 我重新接手时，该怎么检查三个人的完成情况

### 先检查王天一

我应该重点看：

- 他是否只改了 `backend/`
- 他有没有保护住旧接口
- 他有没有新增真正独立的新后端能力
- 他测试有没有继续跟上

重点看：

- `backend/app/api/`
- `backend/app/services/`
- `backend/tests/`
- `backend/BACKEND_ROUND2_DELIVERY.md`

如果他做得不好，常见问题通常是：

- 旧接口被改坏
- 新能力和旧结构缠在一起
- 测试没补
- 想做 AI，结果全是半成品

如果出现这些问题，我应该怎么补：

- 优先保住旧接口
- 把新能力收回成独立 service / route
- 补最少可用测试
- 放弃不稳定的大跨度重构

### 再检查马源胜

我应该重点看：

- 他是否真的只改了 `knowledge_base/` 和 `data/processed/`
- 他有没有把素材做得更结构化
- 他有没有做出“检索卡片 / 评测集 / 场景目录”这类真正好用的资料

重点看：

- `knowledge_base/faq/`
- `knowledge_base/manuals/`
- `knowledge_base/glossary/`
- `data/processed/`
- `data/processed/DATA_AI_ROUND2_DELIVERY.md`

如果他做得不好，常见问题通常是：

- 又写了一堆泛泛的文档
- 资料不够结构化，后面很难被程序利用
- 说要做 RAG 素材，结果只是更多普通 md

如果出现这些问题，我应该怎么补：

- 优先保留结构化程度高的文件
- 把空泛说明压缩掉
- 把“问题 / 关键词 / 应命中文件 / 预期回答”这类结构补出来

### 最后检查周由

我应该重点看：

- 他是否真的只改了 `frontend/`
- 他有没有把图表做成正式图表
- 他有没有继续围绕真实接口增强
- 他的说明里写的东西是不是代码里真的有

重点看：

- `frontend/src/views/`
- `frontend/src/components/`
- `frontend/src/lib/api.js`
- `frontend/package.json`
- `frontend/FRONTEND_ROUND2_DELIVERY.md`

如果他做得不好，常见问题通常是：

- 图表看起来花，但不稳
- 引入了太多依赖
- 又出现“说明大于实现”
- 页面样式变花了，但信息结构没更强

如果出现这些问题，我应该怎么补：

- 优先保留接了真实接口的真图表
- 把不稳的炫技效果拿掉
- 保住数据浏览、分析和问答三块演示链路

## 我第二次整合时应该怎么动手

按这个顺序最稳：

1. 先确认第一次整合后的基线还在
2. 先检查并吸收 `backend/`
3. 再检查并吸收 `knowledge_base/` 和 `data/processed/`
4. 最后检查并吸收 `frontend/`
5. 整合完后自己跑一轮中间验收

不要反过来先看前端，不然还是容易被表现层带偏。

## 第二次整合完以后，仓库应该变成什么样

理想状态大概是：

### 1. 项目层面

- 还是一个统一仓库
- 不是三份增强包简单堆在一起
- 第二轮增强已经被整合成一致的演示口径

### 2. 后端层面

- 旧接口稳定
- 新增能力清楚
- 测试继续变强

### 3. 前端层面

- 至少一部分分析已经是真图表
- 交互更完整
- 演示感更强

### 4. 数据与知识层面

- 知识库更适合检索
- 数据增强方向更清楚
- 问答评测和场景目录更成体系

## 我第二次整合完后，应该做的最后一件事

不要立刻开第三轮。

先自己做一轮第二次整合验收，最少确认：

1. 后端还能跑
2. 前端还能 build
3. 数据浏览还能筛选和导出
4. 分析页至少有真图表或更强的正式展示
5. 问答区还能返回结构化回答和引用
6. 新增知识资料都已经入库
7. 整个仓库依然讲得通

如果这 7 件事都成立，那第二次整合才算基本成功。
