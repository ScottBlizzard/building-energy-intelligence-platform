# 29 · 系统数据合理性与计算逻辑验收规范

> 用途：这是"做数据 / 改逻辑"的**强制验收标准**。任何新增、修改底层数据或分析逻辑，都必须逐条对照本规范的"合理区间 / 计算口径 / 红线"自检，越界即视为不合格。
> 来源：基于 2026-06-11 对数据集的只读审计。数据集已于 2026-06-11 重建为**上半年统一模型**（2026-01-01 ~ 2026-06-01，4864 条，4 栋楼原始标签 + 28 个下游合成设备标签；供暖+制冷双峰季节曲线）。下列区间为重建后实测。
> 关联：`docs/28`（ROI 方法学重构）、`docs/19-SEE`（经济评价）。

---

## 0. 总纲：四层质量模型

| 层 | 名称 | 关键问题 |
|---|---|---|
| L0 | 底层原始数据 | 每个字段物理/统计合理、字段间自洽、随时间趋势合理 |
| L1 | 单功能计算口径 | 每个派生指标（异常/风险/损失/SLA/ROI/预算）算法本身有依据、可复算 |
| L2 | 跨功能对齐 | 同一数字在各处含义一致；上游喂下游连贯；前端表述不误导 |
| L3 | 业务闭环 | 操作真实改变未来（干预、关单→预算、时间机器） |

**修复必须自下而上：L0 → L1 → L2 → L3。** 上层矛盾的根因常在下层。

## 1. L0 · 底层原始数据

### 1.1 字段合理区间（基于实测，新增数据须落在同量级）

| 字段 | 实测区间 | 中位 | 合理性约束 / 红线 |
|---|---|---|---|
| electricity_kwh | 158–690 | 298 | 楼层/区域级 3 小时电耗（690 为异常尖峰）；**红线：≤0、>1000** |
| water_m3 | 7.2–12.2 | 9.3 | 红线：≤0 |
| hvac_kwh | 51–211 | 96 | **必 < electricity**；红线：≥electricity |
| cooling_load_kwh | 149–585 | 289 | 红线：>0 却 hvac=0 |
| chilled_water_supply_temp_c | 6.0–7.5 | 6.8 | 供水温；红线：供≥回水温 |
| chilled_water_return_temp_c | 10.9–12.8 | 11.9 | 供回温差应 3–7℃ |
| environment_temp_c | −2.2–33.4 | 11.9 | 冬低初夏高，跨冬季至初夏连续（冬季供暖季为负低温） |
| humidity_rh | 35–70 | 49.7 | 红线：<0 或 >100 |
| occupancy_density_per_100m2 | 0.9–95 | 37.6 | 寒假 1–2 月偏低；红线：<0 |
| equipment_status | normal / abnormal | — | abnormal 占比 7.2%（350/4864），建议 5–10% |

### 1.2 自洽性约束（必须全部成立，当前实测均通过 ✅）
- electricity > 0、hvac ≥ 0、hvac < electricity。
- cooling_load > 0 时 hvac > 0。
- COP = cooling_load / hvac 落在 **2.2–4.0**（实测 2.31–3.24，中位 3.0）；红线：>6 或 <0.5。
- 供水温 < 回水温。

### 1.3 趋势约束
- **月度电耗呈"供暖+制冷"双峰**：实测均值 1 月 331 →（供暖峰）2 月 318 → 3 月 314 → 4 月 292（过渡谷）→ 5 月 303 → 6 月 325（制冷峰）；环境温度 4.1→27.3℃ 平滑上升。验收：冬季 1–2 月与夏季 6 月为高位，4 月为最低过渡谷；不得出现月际气温跳变（>10℃/月边界）。
- 楼栋差异合理：科研实验楼D（410）> 图书信息楼C（339）> 综合教学楼A（263）> 行政办公楼B（235）。
- 日内 8 个采样点（每 3 小时一条），白天高于夜间；1–2 月寒假占用率显著低于学期月份（供暖仍运行，占用≠负荷）。

### 1.4 🔴 L0 已知关键问题：电耗是"区域级"却被当"设备级"
- 证据：FCU-A-1F-02 单日电耗 190–410 kWh，而单台风机盘管（2.5 kW）日上限仅 ~60 kWh；各设备类型每条均值几乎相同（风机盘管 287 / 冷水机组 321 / AHU 314 / 冷却塔 286），若为单台设备则物理上不可能。
- 影响：`electricity_kwh` 实为区域聚合量、`equipment_id` 是代表性标签。**做异常演示没问题，但 ROI 把"按设备类型求和的区域电耗"当成"该设备能耗"，是 ROI 一切错误的总根源（见 §3.1）。**
- 整改方向（任选其一并写明口径）：
  1. 数据补充"设备额定功率/台数"，ROI 用设备级能耗（额定功率×运行时数或分项计量）；
  2. 明确声明分析为"区域级节能潜力"，投资与节省统一按区域口径，不再按单台铭牌。

## 2. L1 · 派生指标计算口径

### 2.1 异常判定 `is_anomaly` ✅（已按 L1 重构）
口径（`analysis_service._add_operational_dimensions`）：
- **基线按「建筑 × 时段(hour)」分组**，每组约 150 个日采样，统计稳定；阈值 `upper_bound = 时段均值 + 2σ`（`_ANOMALY_SIGMA=2.0`，正态下单侧≈2.3%）。消除了白天/夜间混算导致的误判。
- `is_anomaly = 设备状态异常 OR 电耗>upper_bound OR COP<2.2`。`night_high_load` 退化为"夜间且超时段阈值"的标注子集（仅用于异常原因文案，不再是更松的独立规则）。
- COP 阈值全系统统一 `_COP_TARGET=2.2`（与 §2.3 一致）。
- **实测分析异常率 7.2%（351 条 / 4864）**，较旧的 18% 大幅收敛；其中设备状态异常 350 条、低 COP 0%（本数据集 COP 普遍达标）。
- 红线：异常率显著回升至 >12% 即视为阈值/基线被改松。

### 2.2 风险分 `risk_score`（0–100）✅（已按 L1 重构）
口径（`analysis_service._business_impact`，**透明可加、每信号只计一次、无基础分、权重和=100**）：
- `状态故障 _RISK_W_STATUS=30`（equipment_status≠normal）。
- `超耗幅度 _RISK_W_OVERUSE=35 × clamp01(超出时段期望相对幅度 / 0.35)`——电耗比该时段期望均值高 ≥35% 即占满。
- `COP 缺口 _RISK_W_COP=20 × clamp01((2.2−COP)/2.2)`（本数据集普遍≈0）。
- `夜间高负荷 _RISK_W_NIGHT=15`（off-hours 且超阈值）。
- 不再有"规则数×13 + 单独 +12/+8/+6"的重复计分；不再有基础分 30。
- 严重度按分位标定：**高≥70（≈前 10%）/ 中≥45 / 低**，对应 SLA 8h/24h/72h。实测分布 高≈10% 中≈68% 低≈21%。
- 红线：同一信号被计两次；或出现基础分使量程不从 0 起。

### 2.3 浪费电量 `wasted_kwh` ✅（已按 L1 重构）
口径（取 MAX，不求和，避免同一度电计两次）：
- `over_expected = max(0, 电耗 − 时段期望均值)`——直接可测的超额电量。
- `cop_waste = max(0, hvac − cooling_load / 2.2)`（仅 COP<2.2 时）——**物理推导**：以目标 COP 运行所需电输入为 `cooling_load/2.2`，超出即浪费；替换旧的经验系数 0.12。
- `wasted_kwh = max(over_expected, cop_waste)`。
- 删除了旧的 0.25 / 0.05 / 0.03 经验系数与历史阈值混用；COP 阈值统一为 2.2。
- 实测 `wasted_kwh` 中位 ≈ 80 kWh，无异常为 0 的高风险记录。

### 2.4 损失 / 碳 / 可回收 / SLA ✅
- `损失 = wasted_kwh × 0.82 元/kWh`、`碳 = wasted_kwh × 0.5703 kg/kWh`：电价、碳因子（生态环境部 2022 全国电网平均）为真实值，换算正确。
- `可回收 = 损失 × _SAVING_CAPTURE_RATE(0.65)`：保守经验回收率（运维难一次性根治），已在代码注明。
- `SLA = 高 8h / 中 24h / 低 72h`：分级合理。
- 红线：高风险（≥70）却损失=0 → 当前 0 例 ✅。

## 3. L2 · 跨功能对齐

### 3.1 ROI 改造分析 ✅（已按 docs/28 重构）
- 基线电耗一律用**实测年化**；`has_real_data=False`（该楼无此设备/电耗=0）**不产出方案**，从源头消除"回收期 0 年""对不存在设备照算""各楼雷同"。
- 投资与节省**同规模口径**：投资按该楼该类型台数放大，与全类型电耗一致；删除 `max(实测, 铭牌)` 兜底。
- 删除"节省≤0 强塞 15%"兜底——净节省≤0 如实输出"不推荐"。
- 投资改**增量法**：高效更换只算高效差价、变频/智能/综合为改造直接投入。
- 节能率 = 措施基准节能率 × 能效差调节（COP 达标只给下限 0.5），并标注 `saving_basis`。
- 折现率 **8%（社会折现率）**，输出 5/8/10% 敏感性；用 **EAA 等额年值**比较不同寿命方案（compare 主判据：NPV>0 后按 EAA 最大）；新增动态回收期、含碳价情景。
- 顶部方案卡与对比表**共用同一后端结果**；实测不足显示"无数据/数据不足"，COP=0 不判"低效"。
- 红线（保留）：实测=0 不产 ROI；投资/节省同口径；各面板数值一致；回收期 0/0.1 年或超寿命须复核；不同寿命用 EAA。

### 3.2 预算管理 ✅（已重构）
- 预算口径改为 **`预算 = 期望运行水平 × 目标系数(0.97)`**；期望运行水平 = **季节中性日均 × 季节系数 × 30**。
- **季节中性日均**用该楼**全部可见天**去季节后求均值，**独立于被考核月份本身的实测**——打破"用同月实测当预算再比同月实测"的循环，各楼执行率**不再恒等**（实测 4 月 104.7%–107.5%，因楼而异）。
- **季节系数按实测重标**：1-6 月 = 各月日均/全样本日均（1.06/1.02/1.01/0.94/0.97/1.04），方向与数据一致；7-12 月为合理估计（待数据补齐重标）。
- "超额"含义明确且可解释：维持现状约 1/0.97≈103% 略超目标线，处置异常/改造可压回目标内（而非结构性恒定超额）。
- 红线（保留）：基线独立可解释；各楼执行率不恒等；季节系数方向与数据一致；月底预测样本不足须标注。

### 3.3 预算 KPI 评分 ✅（已按闭环五分项重构）
现状：KPI 改为显式 100 分分项，`预算控制 40 + 能效表现 20 + 异常风险/损失 20 + 工单闭环响应 15 + 节能改善/闭环收益 5`。
- 已结算月份用实际执行率计分，不做 30 天归一；进行中月份才使用月末预测。
- 预算状态统一为 `<=100% 健康`、`100%~103% 预警`、`>103% 超预算`。
- 有异常但缺少派单/响应/关闭复核记录时，工单闭环项会扣分，避免“未闭环但满分”。
- 每个月返回 `score_breakdown` 和 `score_reasons`，前端展示分项分数，便于验收和答辩解释。
- 仍待办：若进入生产试点，应按真实历史绩效重新校准各分项扣分斜率（非课程验收红线）。

### 3.4 前端表述 ✅（78 vs 31.2 已澄清）
- 派单卡"得分构成"改为 **加权得分构成（合计 X/100）**，每项标注满分与权重：`风险 31.2/40`、`损失 ./25`、`SLA ./15`、`碳排 ./10`、`复发 ./10`；风险项悬浮提示"原始风险分 78 × 40% = 31.2"。
- 顶部提示明确区分"原始风险分(0-100)"与"按权重折算后的优先级总分"。
- 设备诊断只列该楼实际设备类型，无数据标注、COP=0 判"无数据"不判"低效"。

## 4. L3 · 业务闭环一致性
- 关单 → 预算预测下调（`summarize_budget_impact_from_closures`）：须保证 saved_kwh 来源口径一致。
- 干预/修复 → 未来不再异常；时间机器推进 → 持续产生新异常（已实现）。
- 全系统时间戳跟随沙盘（已实现）。
- **设备级修复口径（已实现）**：修复以 `equipment_id` 为单位——一台设备同一时刻只允许一张未关闭工单（重复派单 `EquipmentAlreadyHandledError`→409）；关单时联动关闭同设备其余未关闭工单（"随同设备修复关闭"）；已登记干预的设备不再产生新工单、并从派单建议中剔除。
- 红线：任何"生成时间/结算时间"出现现实时间；修复后的设备未来仍异常；**已修复设备仍能被反复对历史异常派单**。
