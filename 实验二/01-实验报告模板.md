## 1. 实验背景（Experimental Background）

软件规模度量是软件项目管理、成本估算、工作量评估和软件经济分析的重要基础。在常见的软件规模度量方法中，功能点分析法不依赖源代码行数，而是从用户可识别功能的角度衡量系统规模，因此特别适合在需求分析和设计阶段对软件项目进行估算。

本次实验以我们课程项目“基于大模型的建筑能源智能管理与运维优化系统”为度量对象，采用功能点方法对当前原型系统的软件规模进行分析。该项目定位为面向建筑能源管理和运维支持的 Web 平台，当前包含结构化能耗数据管理、多条件查询、统计分析、异常分析和智能问答等功能。这些功能都具有较强的用户可见性，适合从功能点角度进行拆分和统计。

通过本次实验，我们不仅希望得到课程项目的功能点结果，还希望进一步理解系统边界划分、数据功能识别、事务功能识别以及复杂度判定对最终规模结果的影响，从而把软件工程管理课程中的理论知识与实际课程项目结合起来，为后续工作量估算和项目评价打下基础。

## 2. 实验内容与步骤（Experimental Content and Steps）

### 2.1 测量对象

本次实验的测量对象为本组课程项目“基于大模型的建筑能源智能管理与运维优化系统”。结合当前项目状态，系统属于 `new development project`，即新开发项目，而不是在既有系统基础上的增强型开发项目。

### 2.2 系统边界与测量范围

本次实验中，测量边界定义为面向用户的 Web 平台本身，包括平台提供的数据查询、统计分析和智能运维支持等核心功能。当前建议纳入测量范围的模块包括：

- 能耗数据总览与核心指标展示
- 按建筑和时间范围查询能耗记录
- 按时间粒度进行汇总统计分析
- 建筑间对比分析与 COP 分析
- 异常识别与异常原因展示
- 基于项目知识和运维资料的智能问答功能

当前建议暂不纳入测量范围的内容包括：

- 启动脚本和本地开发脚本
- 持续集成配置文件和仓库协作文档
- 测试辅助文件
- 尚未形成完整用户可识别业务功能的临时占位实现

本次实验中，小组选择 `【待替换：NESMA / IFPUG】` 作为主测量方法，并使用 `【待替换：IFPUG / NESMA / high-level verification】` 作为验证方法。整体测量流程遵循功能点分析的一般步骤：

### 2.5 实验步骤

实验开始后，我们首先阅读了课程项目的需求分析、技术方案、接口契约和当前原型功能说明，对系统的业务范围和用户目标进行了梳理。在此基础上，小组围绕“什么功能属于真正的用户可识别功能”展开讨论，而不是直接按照源码文件数或 API 数量进行机械统计。

随后，我们将候选功能划分为两大类：

- 数据功能，如 `ILF` 和 `EIF`
- 事务功能，如 `EI`、`EO` 和 `EQ`

在确定候选功能之后，我们结合选定的方法对相关 `DET`、`RET` 和 `FTR` 指标进行统计，判断每个功能点的复杂度级别，并计算其对应的功能点值。最后，对所有功能点结果进行汇总，得到课程项目的规模估算结果，并与验证方法的结果进行比较。

### 3.4 结果分析

本次测量结果表明，当前课程项目已经形成了一组较为清晰的用户可识别功能，其中对功能点结果贡献较大的部分主要来自 `【待替换：例如记录查询、统计分析输出、异常展示、智能问答交互】`。在测量过程中，最有争议和最需要讨论的部分主要是 `【待替换：例如系统边界划分、EO 与 EQ 的区分、辅助接口是否应单独计入】`。

从实验过程来看，功能点分析不仅仅是代入公式进行计算，更重要的是对业务目标、用户视角和系统功能边界进行合理解释。因此，最终结果既反映了当前系统的规模，也体现了小组对项目结构和业务功能的理解。

## 4. 实验学习体会（Your Learning Experiences in This Experiment）

`【以下内容必须由每位同学单独改写，不建议四个人完全一样。】`

通过本次实验，我对软件规模度量和功能点分析有了更直观的认识。与单纯观察代码量或页面数量相比，功能点方法更强调用户可识别业务功能，因此更适合用于项目早期的软件工作量估算。

在实验过程中，我体会到最困难的部分并不是公式本身，而是系统边界的确定和功能分类的判断。例如，一个接口是否应被视为独立事务功能，关键不在于它是否作为单独的 API 存在，而在于它是否构成了一个完整的、用户可感知的基本处理过程。这让我认识到，软件工程中的度量工作不仅需要技术理解，也需要对业务需求进行细致分析。
