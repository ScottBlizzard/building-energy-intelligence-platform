## 1. 实验背景（Experimental Background）

软件规模度量是软件工程管理、软件成本估算、工作量评估和软件经济分析的基础。功能点分析法（Function Point Analysis，FPA）从用户可识别功能出发衡量系统规模，不依赖源代码行数，适合在需求分析、设计、迭代验收和项目评价阶段使用。

本次实验依据《实验二：Software Size Measurement》的要求，选择课程项目作为度量对象，先采用 IFPUG 高层级功能点分析方法估算系统规模，再采用 NESMA 估算法进行结果验证。与原报告相比，本更新版不再停留在 2026 年 5 月中旬的早期原型，而是结合当前仓库多次 Git 迭代后的封版系统重新划定边界和功能点。

### 1.1 功能点分析法简介

FPA 将软件功能分为两类：

- 数据功能：包括内部逻辑文件（ILF）和外部接口文件（EIF），反映系统维护或引用的持久化业务数据。
- 事务功能：包括外部输入（EI）、外部输出（EO）和外部查询（EQ），反映用户、外部系统或客户端与系统之间的可识别交互。

在 IFPUG 高层级估算中，可将数据功能按低复杂度、事务功能按平均复杂度处理，得到未调整功能点 UFP；再根据 14 项通用系统特性（GSC）计算值调整因子 VAF，最终得到调整后功能点 FP。

### 1.2 度量对象更新说明

本次度量对象为本组课程项目“基于大模型的建筑能源智能管理与运维优化系统”。当前系统已经从早期的“数据查询 + 统计分析 + 问答”原型，迭代为面向建筑能源运营闭环的 Web 系统。系统主要技术与功能如下：

- 后端：FastAPI，提供 REST API，并通过同一服务层暴露 MCP Server 能力。
- 前端：Vue 3 + Vite，形成“角色登录 / 总览 / 数据浏览 / 统计分析 / 工单中心 / 我的工单 / 预算管理 / 改造分析 / 决策报告 / AI 助手”的工作台。
- 数据：当前样例能耗数据为 4,864 条记录，覆盖 4 栋建筑、2026-01-01 00:00:00 至 2026-06-01 21:00:00，并包含设备编号、设备状态、环境温湿度、人员密度、暖通能耗、制冷负荷等字段。
- 持久化：默认使用 CSV/JSON 文件存储，配置 `DATABASE_URL` 后可切换到 MySQL 持久化后端。
- 智能能力：知识库检索、可选外部大模型问答、MCP Tools/Resources、异常解释和接地式业务问答。
- 业务闭环：时间沙盘、异常识别、风险量化、工单派发、工人接单、现场处理、管理员复核、预算/KPI 反馈、ROI 改造分析和决策建议。

因此，本报告的度量目标不是回顾早期原型，而是估算当前封版系统在用户可见功能层面的软件规模。

### 2.2 确定项目类型

本课程项目仍属于新开发项目（New Development Project）。虽然系统经过多次 Git 提交迭代，但这些提交属于同一课程项目从需求、原型到封版交付的范围扩展，并非对外部既有应用的增强维护。因此，本报告采用“新开发项目的阶段性更新测量”口径，并根据当前封版范围重新识别功能点。

### 2.3 确定系统边界

本次测量边界定义为：面向用户、演示者和 AI 客户端提供建筑能源管理能力的应用系统本身，包括前端 Web 工作台、FastAPI 后端、持久化业务状态、知识库检索、可选外部 LLM 调用封装以及 MCP Server 对外暴露的同源业务能力。

纳入测量范围的功能包括：

- 建筑能耗数据读取、筛选、统计和导出；
- 建筑总览、时间趋势、建筑对比、COP、楼层区域和设备运行分析；
- 异常检测、异常原因解释、风险评分、节能建议和运营日报；
- 角色登录、管理员看板、工人工作台和权限边界；
- 工单创建、派单、接单、提交、复核、驳回、忽略、附件和时间线；
- 时间沙盘、维修干预、反事实情景对比；
- 预算生成、预算调整、预算执行分析和年度 KPI；
- ROI 改造审计、单方案经济评价、多方案对比和改造候选推荐；
- 知识库检索、AI 助手问答、外部模型可选调用；
- MCP Tools 与 Resources 暴露的数据查询、分析、业务闭环和问答能力。

不纳入测量范围的内容包括：

- 启动脚本、安装脚本、打包脚本和本地开发辅助脚本；
- 课程协作文档、PPT 说明、演示录屏说明等过程材料；
- 自动化测试代码本身；
- 同一业务功能在 REST API、MCP 和前端之间的重复暴露。对于 REST 与 MCP 共享服务层的情况，本报告按用户可识别业务功能计一次，不按接口数量重复计数。

### 2.4 识别功能类型

本报告采用 IFPUG 高层级估算法。数据功能按低复杂度计权，事务功能按平均复杂度计权。复杂度权重如下：

### 3.6 VAF 计算

根据 IFPUG 方法，对 14 项通用系统特性进行评估。

| 序号 | 通用系统特性 | 分值 Ai | 理由 |
| ---: | --- | ---: | --- |
| 1 | 数据通讯 | 5 | 系统同时提供 REST、MCP、可选 LLM 调用和前后端通信 |
| 2 | 分布式数据处理 | 4 | 前端、后端、持久化层、MCP 和外部模型逻辑分离 |
| 3 | 性能 | 3 | 多图表、多分析接口和缓存机制要求较好的响应性能 |
| 4 | 高使用强度配置 | 2 | 课程演示系统而非生产高并发系统 |
| 5 | 事务率 | 3 | 工单、预算、沙盘和问答产生较多交互事务 |
| 6 | 在线数据输入 | 4 | 用户可在线提交工单、处理记录、预算、沙盘控制和问题 |
| 7 | 最终用户效率 | 4 | 角色化工作台、图表联动和自动建议提升用户效率 |
| 8 | 在线更新 | 4 | 工单、预算、沙盘和维修干预均可在线更新 |
| 9 | 复杂处理 | 5 | 包含异常检测、风险评分、预算预测、ROI、反事实和 AI 问答 |
| 10 | 可重用性 | 4 | REST 与 MCP 复用服务层，分析服务可被多入口调用 |
| 11 | 安装易用性 | 2 | 提供脚本和文档，但仍需 Python、Node、可选 MySQL/LLM 配置 |
| 12 | 运行易用性 | 3 | 具备一键检查和演示重置，但多服务协同仍需管理 |
| 13 | 多站点 | 2 | 主要面向本地演示，可迁移到不同开发环境 |
| 14 | 促进变更 | 3 | 模块化目录、接口契约和文档体系支持迭代 |
| **合计 TDI** |  | **48** |  |

值调整因子为：

```text
VAF = 0.65 + 0.01*TDI = 0.65 + 0.01*48 = 1.13
```

### 3.7 最终功能点结果

```text
FP = UFP * VAF = 254 * 1.13 = 287.02 ≈ 287 FP
```

因此，按当前封版系统范围和 IFPUG 高层级估算法，本课程项目的软件规模估算为 **287 FP**。

### 3.8 结果分析

从 UFP 结构看，当前系统规模主要集中在外部输出（EO），共 24 项、120 FP。这与项目演进方向一致：系统不只是记录能耗数据，而是持续输出统计分析、异常解释、工单建议、运营日报、预算/KPI、ROI 和智能问答结果。

ILF 由原报告的 4 项扩展为 8 项，主要原因是当前项目已经形成真实业务闭环，需要维护工单、预算、沙盘干预、用户角色等内部逻辑数据。EI 也从早期的数据导入、查询和问题输入，扩展为围绕工单生命周期、预算维护和时间沙盘控制的在线输入。EQ 主要保留为不产生复杂派生计算的检索型功能。

与原报告 105 FP 相比，新结果显著增加，原因并不是计算方法改变，而是系统范围实质扩大：从“能耗查询分析原型”扩展为“建筑能源运营闭环系统”。这也符合实验指导书中关于新开发项目在开发推进过程中应更新测量并捕获范围扩展的说明。

### 3.9 NESMA 估算法验证

实验指导书要求使用另一种 FPA 方法验证结果。本报告采用 NESMA 估算法进行交叉验证。验证时沿用高层级组件权重，但采用更严格的输入口径，并不计算 VAF。

在 NESMA 严格口径下，“智能助手问题输入”主要作为生成 EO 的控制信息，不单独计为 EI。因此 EI 从 10 项调整为 9 项，其余组件保持不变：

| 功能类型 | IFPUG 主方法 | NESMA 验证方法 | 差异 |
| --- | ---: | ---: | --- |
| ILF | 8 | 8 | 无 |
| EIF | 2 | 2 | 无 |
| EI | 10 | 9 | 智能问答问题输入作为 EO 控制信息处理 |
| EO | 24 | 24 | 无 |
| EQ | 7 | 7 | 无 |
| UFP | 254 | 250 | -4 |
| VAF | 1.13 | 不适用 | NESMA 估算法不使用 VAF |
| 最终 FP | 287 | 250 | -37 |

NESMA 计算如下：

```text
FP_NESMA = 8*7 + 2*5 + 9*4 + 24*5 + 7*4
         = 56 + 10 + 36 + 120 + 28
         = 250 FP
```

两种方法的结果差异为：

```text
(287 - 250) / 287 ≈ 12.9%
```

差异主要来自两个方面：

- IFPUG 主方法通过 VAF 将当前系统的数据通讯、复杂处理、在线更新、复用性等非功能影响纳入规模调整；
- NESMA 验证方法对控制型输入更严格，并直接使用未调整功能点。

两种结果差异约 12.9%，处于功能点估算实践中可接受的波动范围内。因此，可将 **250 FP** 视为当前系统的保守规模下界，将 **287 FP** 视为考虑非功能复杂度后的主估算结果。

## 4. 实验学习体会（Learning Experiences）

### 4.1 我所学到的

通过本次更新测量，我更加清楚地认识到功能点分析不是一次性表格填写，而是随着项目边界变化不断修正的管理活动。早期报告中，系统主要被理解为数据查询、统计分析和问答工具；但当前系统已经包含角色协作、工单闭环、预算反馈、ROI 决策和 MCP 对外能力。如果仍沿用早期功能点数量，就会低估项目真实规模，也会影响后续工作量、进度和经济评价。

本次重新识别时，我尤其体会到“用户可识别功能”和“接口数量”不是一回事。REST API、MCP Tool 和前端按钮可能对应同一项业务功能，不能机械重复计数；反过来，同一个页面上呈现的预算、ROI、反事实和工单优先级，如果背后处理逻辑和业务目的不同，也应拆分为不同 EO。这个过程让我对 FPA 中边界划分、事务合并和复杂度判断有了更实际的理解。

### 4.2 个人贡献

在本次实验中，我主要负责数据和 AI 相关功能的梳理与复核。我重点检查了样例能耗数据、数据字典、知识库、RAG 检索素材、外部 LLM 依赖和数据质量口径，判断哪些属于系统内部维护的数据功能，哪些只是外部引用资料或服务。

在更新报告时，我参与确认了能耗数据、建筑设备逻辑台账、知识库、预算、工单、沙盘状态等 ILF 的边界，并协助区分知识库检索、AI 问答响应、异常解释和 MCP 暴露能力在功能点统计中的归属。通过这些工作，我更加理解了数据生命周期对 ILF/EIF 判断的影响。

### 4.3 个人感想

原报告中的 105 FP 对早期原型是合理的，但当项目演进为完整业务闭环后，功能规模自然上升到约 287 FP。这一变化让我直观看到软件规模度量对项目管理的价值：它不仅给出一个数字，更能反映系统功能复杂度、团队协作范围和后续维护压力。

作为负责数据与 AI 素材的成员，我也意识到数据资产不是报告中的背景材料，而是系统功能规模的重要组成部分。知识库、数据字典、预算口径、设备状态和异常解释规则，都会影响用户可见功能，也会影响功能点估算。未来在参与更复杂的软件项目时，我会更早把功能点识别纳入需求和迭代管理过程，避免等到项目封版后才发现规模口径已经发生较大变化。
