# 第一次任务统一提交要求

这份要求三个人都一样，而且这次要按“独立成果包”来交。

## 1. 提交时必须说明的内容

每个人交回来的时候，都必须把下面 5 件事说清楚：

1. 你改了哪些文件
2. 你新增了哪些文件
3. 你这轮完成了什么
4. 你这轮没完成什么
5. 你觉得下一步谁需要接你这里

## 2. 提交内容应该长什么样

最理想的是：

- 一份你自己负责目录内的成果
- 一份简短说明
- 不夹带跨目录修改

说明内容建议就按下面这个模板写：

```text
姓名：
你负责的目录：
本轮完成：
修改文件：
新增文件：
未完成项：
需要配合的地方：
注意事项：
```

## 3. 这轮尽量不要做的事

- 不要顺手改别人目录里的东西
- 不要顺手改公共文档里的规则
- 不要大规模格式化全仓库文件
- 不要为了“做得更完整”去重构整个项目
- 不要自己改字段名、接口路径再不说
- 不要提交“顺手帮别人也改了一点”的混合成果

## 4. 如果你卡住了，应该怎么做

不要拖到最后一天再说。

你应该直接告诉许奕：

- 卡在哪
- 是技术问题、数据问题还是接口问题
- 你希望谁配合你

## 5. 第一轮验收重点

第一轮不是看谁写得最多，而是看：

- 有没有按边界做事
- 产出能不能作为独立包接进统一仓库
- 有没有给第二轮留下清楚的接口

## 6. 给三个人的统一建议

- 第一轮做“基础版”比做“半成品完整版”更重要
- 把边界守住，比多写一点代码更重要
- 做出来的东西要方便整合，不要只方便自己本地运行
- 你交回来的时候，最有价值的信息不是“我写完了”，而是“我现在做到哪、下一步怎么接”

## 7. 许奕整合时最希望看到的样子

理想情况是：

- 王天一交一个只改了 `backend/` 的后端包
- 马源胜交一个只新增 / 修改了 `knowledge_base/` 和 `data/processed/` 的素材包
- 周由交一个只改了 `frontend/` 的前端包

如果你交回来的成果已经长成这个样子，整合成本会非常低。
