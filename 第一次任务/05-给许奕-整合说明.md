这份文件是写给“未来重新接手这个仓库的我”的。

如果对话上下文突然没了，或者我被重新拉进这个项目，我不要先凭记忆乱判断，而是按这份说明重新熟悉仓库、重新判断三个人的完成情况、再做第一次整合。

## 先记住这次第一轮协作的真正目标

第一轮不是让三个人一起混着改项目，而是：

- 王天一交一个独立的 `backend/` 成果包
- 马源胜交一个独立的 `knowledge_base/` 和 `data/processed/` 成果包
- 周由交一个独立的 `frontend/` 成果包
- 最后由我统一做第一次整合

所以第一次整合的目标不是“全部做完”，而是把这三份独立成果接回同一个仓库，并让整个项目达到**可继续进入第二轮**的状态。

## 我心里的理想样子应该是什么

第一次整合完成后，理想状态不是“项目终版完成”，而是下面这样：

### 后端理想状态

- `backend/` 仍然结构清楚，没有被乱改成一团
- 已有 API 路径保持稳定
- 至少当前这些接口仍然存在并可理解：
  - `GET /api/v1/overview`
  - `GET /api/v1/dataset-meta`
  - `GET /api/v1/buildings`
  - `GET /api/v1/records`
  - `GET /api/v1/analytics/time-summary`
  - `GET /api/v1/analytics/building-comparison`
  - `GET /api/v1/analytics/cop-ranking`
  - `GET /api/v1/analytics/anomalies`
  - `GET /api/v1/analytics/anomaly-reasons`
  - `POST /api/v1/assistant/query`
- 如果王天一补了导出功能，那它应该是纯后端能力，不要求别人先改别的目录
- 后端测试至少比原来更完整，而不是更少

### 前端理想状态

- `frontend/` 仍然只承担展示和调用接口，不自己发明新的数据契约
- 页面至少还是围绕这 4 个区域：
  - 总览
  - 数据浏览
  - 统计分析
  - 智能问答
- 页面结构更清楚、更好演示
- 即使还有占位，也应该是“结构定了，后面再补”，而不是乱掉

### 知识与数据素材理想状态

- `knowledge_base/` 不再只是占位，而是已经多了可直接引用的知识文档
- `data/processed/` 至少多了一份对样例数据的检查或说明文档
- 马源胜如果做得好，后面问答模块应该已经有“可引用内容”和“可问问题”
- 样例数据主文件和数据字典原则上不应该被他私自改掉

### 全仓库理想状态

- 公共契约没被乱改
- 三个人的成果尽量还是边界清楚
- 仓库依然可读、可解释、可继续推进第二轮

## 重新接手时，我该怎么检查三个人的完成情况

### 先检查王天一的完成情况

我应该重点看：

- 他是否主要只改了 `backend/`
- 他有没有私自改 API 路径
- 他有没有补强测试
- 他有没有把后端改得更稳，还是只是随手重构了一堆

我具体应该重新看这些：

- `backend/app/api/`
- `backend/app/services/`
- `backend/tests/`
- 如果有新增：
  - `backend/app/api/routes/export.py`
  - `backend/app/services/export_service.py`
  - `backend/tests/test_export_endpoint.py`

如果他的完成质量不好，常见问题通常是：

- 改了很多结构，但没有提升稳定性
- 接口返回结构乱了
- 测试没有跟上
- 越界改了别的目录

如果出现这些问题，我应该怎么补：

- 把接口路径和字段恢复到契约
- 把 route / service 的基本边界整理回去
- 补最少可用测试
- 删除或放弃无价值的大重构

### 再检查马源胜的完成情况

我应该重点看：

- 他是否真的只改了 `knowledge_base/` 和 `data/processed/`
- 他写的内容是不是可直接引用，而不是泛泛而谈
- 他有没有提供成体系的问题清单 / 术语解释 / 运维建议
- 他有没有越界去动样例数据主文件和数据字典

我具体应该重新看这些：

- `knowledge_base/manuals/`
- `knowledge_base/glossary/`
- `knowledge_base/faq/`
- `data/processed/`

重点留意这些可能新增的文件：

- `data/processed/data_quality_report_round1.md`
- `knowledge_base/manuals/anomaly_diagnosis_guide.md`
- `knowledge_base/manuals/equipment_maintenance_playbook.md`
- `knowledge_base/glossary/metrics_and_rules.md`
- `knowledge_base/faq/qa_bank_round1.md`

如果他的完成质量不好，常见问题通常是：

- 写了很多空话
- 内容和项目字段、模块脱节
- 问题库不适合展示
- 越界改了样例数据或数据字典

如果出现这些问题，我应该怎么补：

- 保留有用的知识文档骨架
- 把空泛内容改成“问题 -> 判断 -> 建议”
- 重新整理 FAQ，使之贴合现有系统接口和页面
- 撤掉越界改动，保留说明类成果

### 最后检查周由的完成情况

我应该重点看：

- 他是否真的只改了 `frontend/`
- 页面结构是不是更清楚了
- 有没有新增合适的组件
- 有没有偷偷改对后端字段的假设

我具体应该重新看这些：

- `frontend/src/views/`
- `frontend/src/components/`
- `frontend/src/lib/api.js`
- `frontend/src/styles.css`
- `frontend/package.json`

如果他的完成质量不好，常见问题通常是：

- 页面看起来花，但结构更乱
- 组件拆分没有逻辑
- 接口字段自己改名适配
- 为了展示方便写死了一堆本地假数据

如果出现这些问题，我应该怎么补：

- 优先保留结构清楚的组件
- 把写死假数据的地方换回真实接口响应
- 把页面重新收回到 4 个主区域
- 把样式问题留到第二轮，先保联调

## 我重新接手时应该怎么动手整合

按这个顺序最稳：

1. 先确认仓库基线还在
2. 先检查并吸收 `backend/`
3. 再检查并吸收 `knowledge_base/` 和 `data/processed/`
4. 最后检查并吸收 `frontend/`
5. 整合完后自己跑一轮中间验收

不要反过来先看前端，不然很容易被表现层带偏。

## 如果三个人的成果“不够好”，我要重点补哪里

第一次整合时，我不应该试图把所有问题都一次补平，而是优先补这些：

1. 公共契约一致性
2. 后端接口还能不能稳定支撑前端
3. 前端是否还能围绕真实接口展示
4. 知识库内容是否至少可用
5. 仓库结构是否仍然清晰

我不应该优先补的，是这些：

- 特别细的美化
- 过度重构
- 第二轮才值得做的复杂功能

## 第一次整合完以后，仓库应该变成什么样

第一次整合结束后，仓库理想上应该达到这个状态：

### 1. 项目层面

- 这是一个统一的、还能继续开发的仓库
- 不是三份独立作业堆在一起
- 也不是为了整合而把结构搞烂

### 2. 后端层面

- 后端接口路径清晰、结构稳定
- 当前 API 契约基本还成立
- 测试至少比原来更完整

### 3. 前端层面

- 前端能打开
- 前端能围绕真实接口展示核心内容
- 前端 4 个主区域逻辑连贯

### 4. 数据与知识层面

- 样例数据仍然是主基线
- 知识库新增内容已经入库
- 数据检查和问答素材已经可见

### 5. 管理层面

- 我已经知道第二轮该怎么继续分工
- 我已经知道第一次整合暴露了哪些问题
- 我可以基于这些问题继续布置第二轮任务

## 我第一次整合完后，应该做的最后一件事

不要立刻开第二轮。

先自己做一轮中间验收，最少确认：

1. 后端能不能跑
2. 前端能不能打开
3. 数据浏览是不是还能看到记录
4. 分析页是不是还能看到数据
5. 问答区是不是还能返回内容
6. 新增知识文档是不是都在
7. 整个仓库是不是还讲得通

如果这 7 件事都成立，那第一次整合就算基本成功。
